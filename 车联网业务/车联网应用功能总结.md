# 中国电信车联网运营支撑平台 — 应用功能与数据模型总结

> **文档性质**：基于《车联网运营支撑平台_需求规格说明书》（V1.0.1，2024-08-06 草案）与本地模型资产（`车联网模型文档toon`，源自《车联网应用模型_SDM模型》Excel 转 TOON）整理，用于从使用者、数仓建模、产品项目三视角快速对齐范围与数据脉络。  
> **说明**：大屏页面由亿迅实现，指标由感知/位置产品与亿迅确认；部分车辆品牌等字段标注依赖外部号卡资源数据。

---

## 1. 使用者视角：用了哪些电信数据、用在哪、带来什么价值

### 1.1 电信侧数据类型总览

平台依赖的核心网络数据在需求书中明确为 **4G/5G MR（测量报告）**、**4G/5G DPI（深度包检测）详单**，并配合 **工参、地图/路网** 等基础数据做呈现与轨迹拟合。结合模型清单，可归纳为以下几类：

| 数据大类 | 典型来源/形态（模型中的体现） | 主要用途 |
|----------|------------------------------|----------|
| **MR / 覆盖测量** | 4G MRO、5G NR MRO、栅格/小区级 MR 汇聚；45G 位置事实与轨迹 | 轨迹与位置还原、RSRP/弱覆盖判断、覆盖率与电平分布、GIS 打点与路网拟合 |
| **DPI 用户面（xDR）** | HTTP/HTTPS/Video 等 4G DPI；S1U HTTP/HTTPS/Video、5G SA HTTP/HTTPS/Video/Other 等 | 业务识别、流量/时长/次数、网页与视频等 KPI、数据业务“好坏单”与感知评分 |
| **控制面信令** | LTE S1MME、NR N1N2（含原因码配置维表） | 附着/注册、切换、掉线、上下文释放等 **性能与失败定界**，支撑“感知诊断”各环节 |
| **无线与工程参数** | LTE/NR 工参投影、小区维度（cellkey 等） | MR/DPI 与小区、扇区、地图图层关联；基站扇区 GIS |
| **用户与车联网清单** | 车联网用户清单（如 `cfg_iov_usr_list` 类配置）、号卡资源（ODS） | 限定车联网号卡范围、与车辆品牌/终端状态等展示字段关联 |
| **静态编码维表** | S1MME / N1N2 cause 码表 | 失败原因解释、定界展示 |

### 1.2 数据 → 平台功能映射（按需求书中的功能模块）

- **轨迹感知概览（号卡查询、业务概况、感知趋势、出行路段）**  
  - **数据**：DPI 话单聚合得到高频业务 TOP、4G/5G 分流与驻留；MR/覆盖类指标得到路段级无线感知；号卡资源提供车辆品牌、产品状态等展示字段。  
  - **用途**：一眼看清“谁在用什么业务、无线和数据感知好不好、按路段怎么变”。

- **路段感知分析卡片（网络分析 / 应用感知 / 感知诊断）**  
  - **数据**：小区占用数、MR 覆盖率、RSRP 均值与趋势来自 MR/覆盖汇聚；TOP 应用来自 DPI 按 app 汇聚；附着成功率、掉线率、切换成功率等来自 **S1MME/N1N2** 与综合指标表；诊断高亮依赖端到端指标与原因码。  
  - **用途**：从“路段”粒度把无线、数据业务、信令问题拆开，支撑运营人员快速判责。

- **用户感知详情回溯**  
  - **数据**：完整暴露控制面、用户面字段：网络类型、接口类型、时间、位置、RSRP、流量、时延、HTTP 状态、DNS、TCP、视频卡顿、游戏时延等（需求书列表与 DPI/MR/信令模型一致）。  
  - **用途**：对话单级问题追根溯源，并与 GIS 上 MR 轨迹联动。

- **GIS 可视化**  
  - **数据**：轨迹与停驻点（Flink/轨迹类事实与出行表）；4G/5G MR 采样点（拟合纠偏后经纬度）；占用基站工参与扇区图层。  
  - **用途**：可点选 MR 看小区连线、弹窗看 RSRP/RSRQ/SINR/弱覆盖等，实现“看得见”的投诉与优化支撑。

- **全国/省级运营大屏**  
  - **数据**：在区域维度汇聚的覆盖、数据感知评分、附着/掉线/切换、4G/5G 电平分段占比、质差路段占比等（应用层如区域共享指标表）。  
  - **用途**：管理者看全网/省/市体验与趋势；指标通过数据共享给亿迅做大屏（需求书约定）。

- **数据管理（接入、清洗、共享）**  
  - **数据**：工参、地图、MR、DPI 及路网/POI 清洗结果。  
  - **用途**：保证轨迹算法与展示底图一致、指标可对外共享。

### 1.3 对车联网平台的“赋能”表述（面向运营人员）

1. **把网络专业数据翻译成运营语言**：将 MR、DPI、信令转为覆盖评分、数据业务评分、TOP 应用、路段级体验，降低对无线参数的依赖。  
2. **时空闭环**：从全国/省视图 → 单用户路段 → 单次会话/话单回溯，同一套指标体系贯穿。  
3. **定界能力**：用户 / 无线 / 核心网 / SP 多环节指标与原因码，支撑需求中的“感知诊断”与红色高亮。  
4. **可视化决策**：GIS + 轨迹 + 工参，贴近“车在哪、网怎样、业务怎样”的一体化视图。  
5. **可演进**：需求书提及未来与投诉、工单等打通及机器学习分析，当前数据模型已为多维汇聚与明细并存打下基础。

## 2. 模型设计与数仓开发者视角：建模方式与特点

### 2.1 分层与事实/维度关系（偏星型，局部雪花）

- **整合层（事实明细）**：以 **DPI xDR、MR、S1MME、N1N2** 等为 **事实表**（如 `fact_*dpi*_e2e_*`、`fact_lte_s1mme_e2e_telecom_h`、`fact_5gdpi_n1n2_e2e_telecom_h`、MRO/位置类事实），承载最小粒度事件或测量。  
- **中间层（汇聚事实）**：按 **用户|小区|app|栅格|路段|时间** 等做小时/天级 `aggr_*`，车联网专题在相同链路上增加 `*_iov_*` 裁剪与汇聚，连接 **车联网用户清单**。  
- **维度层**：独立 **编码表**（如 S1MME/N1N2 cause 列表）、工参/地理类维度；部分维度属性 **反打** 进汇聚表以减少关联，整体更贴近 **星型**（多事实表共用时间、地域、用户、小区、app 等一致性维度）。  
- **应用层（DM）**：面向页面的 **宽表/主题表**（如 `dm_iov_*`），将 4G/5G 综合指标、路段、应用、区域大屏等 **指标与维度字段合并**，便于 Doris 等引擎查询；来源上多由 `dm_*_perception_comprehensive_usercell_*` 等 **再聚合或 Full Join** 生成。

**“雪花”体现**：当工参、原因码、号卡资源等以 **独立维表** 维护、通过 key 关联时，属于雪花式规范化；为查询性能，应用层往往已 **扁平化** 为星型宽表。

### 2.2 模型设计特点（结合现有表命名与导航）

1. **专题域清晰**：轨迹分析、数据业务感知、覆盖分析、信令分析、维度配置等在中层表主题域与 `导航_表中文名` 中可追踪。  
2. **4G/5G 对称**：同类业务有并行 LTE 与 NR 表，便于统一 UI 与评分逻辑（如综合感知天表合并 AB 两表）。  
3. **粒度递进**：15 分钟/小时/天、路段/用户/小区/区域多粒度，支撑从明细到看板。  
4. **端到端指标合一**：`dm_lte_perception_comprehensive_usercell_*` / `dm_nr_perception_comprehensive_usercell_*` 等 **“综合感知”** 宽表，把覆盖、DPI、信令类度量合并，对上直接服务用户级、区域级 DM。  
5. **实时与离线并存**：轨迹链路中标注 **Flink 程序** 处理（如 `ods_45gmr_location_h`、`fact_45gmr_user_trips_*`、`dm_user_line_d`），与离线数仓表共同支撑轨迹与出行。  
6. **落地与同步策略**：应用表同步 **Doris**、GIS；部分表 **前置/核心** 集群与生命周期（如 7 天/14 天）在元数据中区分，便于治理与成本。

### 2.3 小结（建模视角）

- **主形态**：**多事实表 + 共享维度** 的 **星型** 为主；维表规范化与多层 `aggr` 链形成 **轻度雪花** 中间层，再 **收敛为应用层宽表**。  
- **特点**：电信 DPI/MR/信令 **同源分层**、车联网 **iov 裁剪**、**4G|5G 双栈**、**端到端综合宽表**、**离线聚合 + 实时轨迹** 组合。

## 3. 产品项目团队视角：平台定位与协作

### 3.1 定位与目标

- **定位**：支撑 **集团网运车联网运营业务** 的运营支撑平台（与 D_Location 等产品版本关联），嵌入 **定制网** 等门户使用（全局导航、翼车智联等）。  
- **目标（需求书）**：高精度轨迹还原；用户感知概览、诊断、详情回溯；并预留全国路网客户级覆盖评估、智驾路径规划等远期能力。

### 3.2 功能边界与协作

| 角色 | 职责侧重 |
|------|----------|
| **德特赛维（感知/位置产品 + 数据）** | 指标定义、算法（轨迹识别、感知打标等）、数据模型与共享、大屏 **指标供给** |
| **亿迅** | 大屏 **页面开发**、部分号卡/车辆展示数据 |
| **中国电信网络运营部** | 需求方与业务管理方 |
| **定制网平台** | 菜单嵌入、权限与单点登录等规范对接 |

### 3.3 用户与约束

- **用户画像**：车联网业务运营人员为主，网络经验不深，依赖平台做快速支撑；**高频使用（如 >5 次/天）、用户量预计 100+**。  
- **约束**：合规与数据安全、容器化部署、集团及厂商技术栈要求；浏览器以 Chrome 为主兼容近年 Firefox/360。

## 4. 建议补充的重要内容（实施与运维强烈相关）

1. **数据依赖与就绪条件**：MR、DPI、路网、工参的 **时延、完整性、分区键** 与车联网号卡清单刷新频率；缺失时的降级策略。  
2. **指标口径白皮书**：覆盖/数据/综合感知评分公式、坏单定义、TOP 应用排名规则（需求书注明由感知产品组确定部分规则）。  
3. **端到端数据质量**：与需求书中“接入端到端数据质量稽核”呼应，**关键表**、**对账规则**、**告警阈值**。  
4. **权限与脱敏**：分权分域、MSISDN/IMSI 等日志与 URL 脱敏、金库/4A 等（需求书安全章节已列基线）。  
5. **性能与容量**：查询响应（如 3s/10s）、导出上限（如 30 万条）、Doris 分区与物化策略与模型 `p_date`/省分区的对齐。  
6. **版本与变更**：需求书 V1.0.1 与模型表 `2024-9-9` 版本差异的跟踪方式（字段增减、任务责任人）。  
7. **演进路线**：投诉/工单对接、机器学习预测、二阶段路网/覆盖/感知按钮等（需求书已标注部分为后期）。
