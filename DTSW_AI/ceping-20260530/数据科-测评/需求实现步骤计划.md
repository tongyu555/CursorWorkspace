# 数据科测评需求实现步骤计划（第一部分与第三部分）

根据《2026年5月18日-数据科-测评.md》及通用提示词要求，针对第一部分（工程能力）和第三部分（执行效率）制定以下初步实现计划。**（由于尚未提供《站址合并算法说明.pptx》与《测评模型.xlsx》，此处仅规划实现步骤框架，需后续补充具体业务细节）**

## 一、前期准备阶段
1. **环境与代码仓库准备**
   - 切换/创建个人 Git 分支，遵循 `工号-姓名拼音` 命名规范（如 `9527-wangwu`）。
   - 在 `ai-capability-test` 仓库对应个人目录下创建规范的文件夹结构：
     - 第一部分：`task1_rdd/src/`, `task1_rdd/test/`, `task1_rdd/target/`, `task1_rdd/docs/`
     - 第三部分：`task3_models/sql/`, `task3_models/screenshots/`, `task3_models/docs/`
   - 下载并解压指定的 Spark 包（2.3.4版本）以供本地测试使用。
   - 确认恒大测试环境（dito数据库）的连接与使用方式。

## 二、第一部分：工程能力 - 站址合并算法 RDD 实现 (task1_rdd)
1. **算法设计与方案输出**
   - 获取并阅读《站址合并算法说明.pptx》（**需用户提供**）。
   - 输出《算法设计与使用文档》，阐述基于距离阈值的简单聚类或联通分量算法的实现思路，并包含算法时间复杂度说明及数据倾斜调优思路。
2. **核心算法代码开发**
   - 使用 Scala 或 PySpark 编写核心算法，存放在 `task1_rdd/src/` 中。
   - *（强制要求）在生成的新增代码行前后添加 AI 代码标记 `Generated By AI Start (Cursor)` 和 `Generated By AI End (Cursor)`*。
3. **单元测试与功能验证**
   - 在 `task1_rdd/test/` 中针对各个 RDD 转换算子编写单元测试代码。
   - 构造小规模测试数据集，验证合并结果准确性、异常拦截能力及执行性能。
   - 输出《功能测试报告》（包含小规模数据合并验证结果）。
4. **编译与结果归档**
   - 若为 Scala 项目，完成编译并在 `task1_rdd/target/` 目录下生成可直接 `spark-submit` 运行的 Jar 包；若为 PySpark，则准备好 `.py` 运行脚本。
   - 完善 `task1_rdd/docs/` 下的《运行测试截图说明文档》及《AI 对话过程文档》。

## 三、第三部分：执行效率 - 开发指定的 10 个模型 (task3_models)
1. **模型业务逻辑梳理**
   - 获取并查阅《测评模型.xlsx》（**需用户提供**），或者基于现有 `ddls` 目录内的 SQL 明确题目要求的10个模型目标。
2. **SQL 脚本开发**
   - 针对 10 个模型，依次开发 Spark SQL 脚本，命名规范为 `model_01.sql` ~ `model_10.sql`。
   - 脚本统一放置于 `task3_models/sql/` 目录下。
   - *（强制要求）在每段生成的 SQL 代码中遵守要求，添加 AI 标记注释，如果是 SQL 可以转换为行级或段级注释，但注意 SQL 中的注释格式（不可使用 Java 行尾注释）*。
3. **测试环境执行验证**
   - 登录“恒大环境-spark服务” dito数据库，执行以上 10 个模型的 SQL 脚本。
   - 确认结果写入目标表或在控制台成功输出。
4. **截图收集与文档归档**
   - 截取 10 个模型运行成功的控制台/日志截图，存入 `task3_models/screenshots/` 目录下（命名如 `run_model_01.png`）。
   - 完善 `task3_models/docs/` 下的《详细设计文档》、《运行测试截图说明文档》及《AI 对话过程文档》。

## 四、最终检查与代码提交
1. **规范性检查**
   - 全局扫描代码，确保 AI 标记注释均按要求添加并合理合并。
   - 核对项目目录结构完全符合测评文档中的规范。
2. **代码提交记录**
   - 按照 Git 开发规范，将各阶段代码分批或最终推送到远程个人分支，确保具备完整的代码提交记录。
