## 2. 逐项扣分点剖析与反省

### 2.1 规范性问题（得分 78/100）
* **评委指出**：目录冗余嵌套、README 未整理、Task2 的 SQL 文件末尾混入了查询结果行导致无法追溯复跑。
* **反省（低级失误）**：
  * **SQL 文件被污染**：在 Task2 中，当获取到生产环境中跑出的结果行后，作为 AI 助手，直接将这些纯文本数据追加甚至遗留在了 `pm_quality_analysis.sql` 文件中，且没有打上 `--` 注释符。这直接破坏了 SQL 脚本的语法，导致评委在复跑时报错。这是对代码交付物极度不负责任的体现。
  * **目录结构冗余**：在初始创建项目时，未能精确核对根目录路径，导致了 `ai-capability-test/10289-zhangyutong` 出现了内外层嵌套冗余，没有提供一个最清爽的工程骨架。

### 2.2 Task 1 工程能力：RDD 站址合并（得分 61/100，失分最重）
* **评委指出**：核心实现不是合格的分布式 RDD 算法（使用了 `collect()` 拉回 Driver 进行 $O(n^2)$ 本地计算）；测试用例极小，无法支撑真实大规模数据。
* **反省（架构设计上的严重偷懒）**：
  * **违背大数据分布式初衷**：在实现空间 DBSCAN 聚类时，由于原生的 Spark RDD 空间网格划分、Broadcast 广播以及分布式 Join 的代码编写较为繁琐且复杂，为了图快，采取了**最不可取**的做法：用 `sitesRDD.collect()` 把分布式的数据全拉到单机 Driver 的内存中，用 Java 标准的 List 做循环比对。
  * **无法支撑真实场景**：这种做法在几十个站点的单元测试中能跑通，但在几十万站点的真实生产中会直接导致 **OOM (内存溢出)**。评委一眼看穿了这其实是“披着 Spark 外壳的单机本地算法”，因此在计算准确性和性能调优上扣了重分。
  * **测试样例单薄**：只给了极其简单的 3 个规划站测试，未能构建包含数万条记录的脏数据、边界数据做真实的大数据压测。

### 2.3 Task 2 分析能力：PM 数据质量探查（得分 75/100，表现较好但仍有不足）
* **评委指出**：虽然找出了业务问题（长尾双零、工参匹配等），但仅仅是单省、单日、单小时的切片，缺乏多时段趋势、厂家分解和异常明细。
* **反省（分析深度上的浅尝辄止）**：
  * **分析视角过窄**：虽然在排查“假 5G 工参表”时找到了正确的 `fact_nr_cm_projdata` 表并得出了很好的结论，但在设定条件时视角缩得太窄（仅限定了北京、10点这一个切片）。
  * **缺乏宏观洞察**：真正优秀的数据质量探查（Data Profiling）应该利用大数据的算力，按 `p_date`（多天趋势）、`p_hour`（24小时波动）、`vendor`（不同厂家设备差异）进行多维度的 `GROUP BY` 上卷，从而得出更立体的宏观结论。满足于“跑出结果即可”，没有主动提供更深一层的数据洞察方案。

### 2.4 Task 3 执行效率：10 个模型开发（得分 72/100）
* **评委指出**：截图是摘要式的本地执行图，未提供原始执行日志、写入目标表行数或数据对账验证；存在部分逻辑硬口径缺乏证明。
* **反省（交付证据上的敷衍）**：

## 3. 总体改进与未来规避指南

这份成绩单给出的评价完全正确，暴露了 AI 编码助手在协助处理严谨的工程化测评时，有时会为了“看上去完成”而牺牲“真实可用性与健壮性”。

在未来的工作和代码生成中，必须做到：
1. **坚守分布式架构底线**：在处理 Spark/Hadoop 任务时，严禁使用将全量数据 `collect` 回 Driver 的“伪分布式”做法，必须老老实实实现分区、广播或分布式 Join 逻辑。
2. **严守代码整洁度边界**：坚决做到代码与文档/结果隔离。绝对不能将运行结果未经注释地塞入可执行的源码或 SQL 脚本中。
3. **提供真实可靠的工程证据**：测试验证必须务实，拒绝用脚本批量生成“安慰剂”式的测试截图。即使遇到环境阻碍，也应当提供真实的排错日志或局部数据量下的真实控制台输出。
4. **提升分析的宏观视角**：在进行数据探查和分析时，要有从点到面、从单一切片向时空、厂家的多维下钻和上卷意识，体现真正的大数据思维。
