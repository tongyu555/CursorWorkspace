### 文档一：项目总体实施方案（管理决策版）

**1. 项目目标**
*   **核心目标：** 将临时取数 SQL 的编写时间从小时级缩短至分钟级。
*   **交付物：** 基于 Cursor 的 AI 辅助开发体系 + AI 驱动的 Hive 知识库。
*   **核心逻辑：** 弃用传统的“查文档-写 SQL”模式，转向“需求描述 - AI 组装逻辑 - 人工审核”模式。

**2. 实施阶段**
*   **第一阶段（准备期）：** 资产数字化。导出 DDL、模型设计和 ETL 生成 SQL，构建 AI 易读的本地知识库。
*   **第二阶段（试点期）：** 配置 Cursor 规则链。建立 `.cursorrules` 和 `@Context` 引用标准，在 2-3 名核心开发中试运行。
*   **第三阶段（提效期）：** 利用 AI 反向补全元数据注释，解决“表看不懂”的问题，全面推广。

**3. 关键成功因素**
*   **知识库质量：** 历史 ETL SQL 必须包含在内，这是 AI 理解业务口径的核心。
*   **SOP 落地：** 开发人员必须养成引用 `@` 文件的习惯。

### 文档二：Hive 资产“AI 化”整理标准（技术实施版）

在将资源导入 Cursor 之前，需按此标准组织文件，以最大化 AI 的理解能力。

**1. 目录结构规范**
```text
/hive-ai-brain/
├── ddl/            # 存放所有表的建表语句 (TableName.sql)
├── logic/          # 存放每张表的生产加工 SQL (了解指标计算逻辑的核心)
├── golden_sql/     # 存放高频、复杂的历史临时取数 SQL 范本
├── docs/           # 存放模型设计 Markdown 或业务指标定义文档
└── .cursorrules    # 全局 AI 行为约束文件
```

**2. 资产预处理（AI 预冲刺）**
*   **自动化任务：** 编写一个脚本，将 Hive Metastore 中的信息导出。
*   **AI 增强：** 针对 `logic/` 中的 SQL，利用 AI 批量提取出“表描述”和“核心字段口径”，并以 README.md 形式存入 `docs/`。

### 文档三：Cursor 核心配置模板（`.cursorrules`）

将以下内容放入项目根目录的 `.cursorrules` 文件中，约束 AI 的输出质量。

```markdown
# Hive SQL 开发助手规则

## 角色定位
你是一个资深的 Hive 开发工程师，精通 Hive SQL 语法、性能优化及业务指标逻辑。

## 知识上下文引用
1. 在生成 SQL 前，请优先检索 `logic/` 目录下对应表的生产 SQL，以确保计算口径（如活跃、订单状态、金额单位）与生产环境一致。
2. 必须参考 `ddl/` 目录下的字段名，严禁幻觉出不存在的字段。

## SQL 编写约束
1. **分区过滤：** 所有的查询必须包含 `dt`（或相关分区字段）的过滤条件。
2. **性能优化：** 
   - 避免使用 `SELECT *`。
   - 大表关联时，注意是否需要 MAPJOIN 提示。
   - 优先使用窗口函数处理 TopN 类需求。
3. **方言要求：** 使用 Hive 标准函数（如 nvl, coalesce, date_format），禁止混用 MySQL 或 Postgres 函数。

## 输出格式
1. SQL 语句前需附带简要逻辑说明。
2. 关键过滤条件需注释说明原因（参考历史逻辑）。
```
