# 深度解析 Apache Doris：基于 MPP 架构的新一代极速分析型数据库

Apache Doris（原百度 Palo）是近年来在开源大数据领域极其火爆的一款 MPP（Massively Parallel Processing，大规模并行处理）架构的分析型数据库（OLAP）。它以“极速易用”著称，能够在亚秒级响应海量数据的复杂查询。

本文将从 MPP 架构的视角出发，深度剖析 Doris 的架构设计、优化改进、独特亮点、应用场景以及运维部署。

---

## 一、 Doris 的 MPP 架构设计与优化改进

传统 MPP 架构（如 Greenplum）虽然解决了海量数据并行计算的问题，但在高并发、实时导入、扩缩容等方面存在明显的短板。Doris 在继承 MPP 核心思想（Share-Nothing、数据分片、计算下推）的基础上，进行了大量针对现代 OLAP 场景的优化。

### 1. 极简的架构组件（FE + BE）

Doris 的架构极其简单，只有两类进程，不依赖任何第三方系统（如 Hadoop、ZooKeeper）：

- **FE (Frontend，前端节点)**：主要负责用户请求的接入、SQL 解析、查询计划生成与调度、以及元数据管理。FE 基于 Java 开发，内部集成了 BDB JE（Berkeley DB Java Edition）来实现元数据的高可用和强一致性。
- **BE (Backend，后端节点)**：主要负责数据的存储和查询计划的执行。BE 基于 C++ 开发，是 Doris 性能的核心引擎。

**MPP 视角的改进**：传统 MPP 往往需要依赖外部组件来管理元数据或协调分布式事务，导致运维复杂。Doris 将所有管理逻辑收敛在 FE，计算和存储逻辑收敛在 BE，实现了真正的“开箱即用”。

### 2. 存算紧耦合与智能的数据分片

在 Doris 中，数据被划分为 Database -> Table -> Partition（分区） -> Tablet（分桶）。

- **Partition (Range/List)**：通常按时间维度进行分区，便于冷热数据分离和快速裁剪。
- **Tablet (Hash)**：在分区内，数据根据指定的 Distributed Key 进行 Hash 打散，形成多个 Tablet。Tablet 是 Doris 数据移动、复制和负载均衡的最小物理单元。

**MPP 视角的改进**：传统 MPP 在扩容时（如增加节点），往往需要对整张表进行极其昂贵的 Rebalance 操作。而 Doris 引入了 Tablet 的概念，扩容时系统会自动在后台以 Tablet 为粒度将数据平滑迁移到新节点，业务完全无感，极大地缓解了 MPP 扩容痛苦的问题。

### 3. 全面的向量化执行引擎

Doris 采用了完全重写的向量化执行引擎（Vectorized Execution Engine）。

- 传统的 Volcano 模型是按行处理数据（Row-by-Row），函数调用开销大，CPU 缓存命中率低。
- Doris 的向量化引擎按列块（Column Block）处理数据，充分利用了现代 CPU 的 SIMD（单指令多数据流）指令集，将计算效率提升了数倍甚至数十倍。

## 二、 Doris 的独特设计亮点

除了基础的 MPP 能力，Doris 还有几个让它在众多 OLAP 引擎中脱颖而出的“杀手锏”。

### 1. 丰富的数据模型（Data Model）

为了适应不同的业务场景，Doris 在底层存储上设计了三种独特的数据模型：

- **明细模型 (Duplicate Key)**：保留所有导入的原始数据，不做任何聚合，适合日志分析、行为打点等需要查询明细的场景。
- **聚合模型 (Aggregate Key)**：在数据导入时，自动对具有相同 Key 的数据进行聚合（如 SUM, MAX, MIN）。极大地减少了存储空间，并在查询时免去了现场聚合的开销，适合报表系统。
- **更新模型 (Unique Key)**：保证 Key 的唯一性。当导入相同 Key 的新数据时，会自动覆盖旧数据。Doris 引入了 Merge-on-Write（写时合并）技术，使得 Unique Key 模型在支持高频实时更新的同时，依然保持了极高的查询性能。



### 2. 强悍的实时写入能力与 Load 机制原理

传统 MPP 数据仓库（如 ClickHouse 早期版本）对高频实时小批量写入非常不友好，容易产生大量小文件导致系统崩溃。Doris 设计了专门的 **Stream Load** 和 **Routine Load** 机制来应对高频实时写入：

- **Stream Load（同步流式导入）**：
  - **设计与原理**：Stream Load 是一种基于 HTTP 协议的同步导入方式，主要用于将本地文件或数据流（如 Flink 实时计算结果）导入 Doris。用户发送 HTTP 请求给 FE，FE 会通过 HTTP Redirect 将请求转发给某一个 BE 节点（作为 Coordinator 协调者）。该 BE 负责接收数据，并根据数据的分布规则（Partition 和 Bucket）将数据分发到其他对应的 BE 节点（Executor 执行者）。
  - **事务机制**：Stream Load 采用两阶段提交（2PC）机制。数据写入完成后，Coordinator BE 会向 FE 发起 Commit 请求，FE 记录元数据后，数据才对用户可见。这保证了导入的原子性。
- **Routine Load（异步例行导入）**：
  - **设计与原理**：Routine Load 是一种自动从 Kafka 等消息队列中订阅数据并持续导入 Doris 的异步机制。用户只需向 FE 提交一个 Routine Load Job（包含 Kafka Topic、Broker 地址等信息）。
  - **调度与执行**：FE 的 Job Scheduler 会将 Job 拆分成多个 Task，并根据 BE 的负载情况将 Task 调度到不同的 BE 节点上执行。BE 节点直接连接 Kafka，消费指定 Partition 的数据，并在本地完成数据的解析、转换和写入。
  - **Exactly-Once 语义**：Doris 内部维护了 Kafka 的 Offset。当 BE 成功将数据写入并向 FE 汇报后，FE 才会更新该 Job 的 Offset 记录。如果 Task 失败，FE 会重新调度并从上一次记录的 Offset 继续消费，从而严格保证了数据的不丢不重（Exactly-Once）。
- **底层存储适配**：底层采用了类似于 LSM-Tree 的结构，通过后台的 Compaction 机制不断合并小文件。这使得 Doris 能够轻松应对 Kafka/Flink 等流式系统每秒数十万条记录的实时高频写入。





### 3. 物化视图（Materialized View）与 Rollup

Doris 支持在原表（Base Table）基础上创建强一致的物化视图（Materialized View）和 Rollup（上卷）索引，以加速复杂查询。

- **Rollup（上卷索引）**：
  - **概念**：Rollup 本质上是 Base 表的一个物理子集或重新排序的视图。它可以包含 Base 表中的部分列，或者改变列的物理排序方式。
  - **原理**：在聚合模型（Aggregate Key）下，Rollup 可以调整聚合的维度（例如 Base 表是按省、市、县聚合，Rollup 可以只按省聚合）；在明细模型（Duplicate Key）下，Rollup 常用于调整前缀索引的顺序，以加速不同维度的点查。
  - **特点**：Rollup 的数据模型必须与 Base 表保持一致（同为明细、聚合或更新模型），且只能减少列或调整顺序，不能进行复杂的表达式计算。
- **物化视图（Materialized View）**：
  - **概念**：物化视图是包含了查询结果的实际物理表。相比于 Rollup，物化视图的功能更加强大。它不仅支持对 Base 表的列进行重新排序和过滤，还支持在明细模型上建立聚合操作（如 SUM, MIN, MAX, COUNT），甚至在新版本中支持多表 Join 的物化视图。
  - **原理与强一致性**：Doris 的单表物化视图与 Base 表是**强一致且同步更新**的。当用户向 Base 表写入数据时，Doris 会在同一个导入事务中，自动将数据路由并更新到物化视图中。只有当 Base 表和物化视图都写入成功时，导入才算完成。用户完全不需要手动维护两套数据。
- **智能路由（透明重写）**：
  - 当用户提交查询 SQL 时，Doris 的优化器（Optimizer）会自动进行查询重写分析。它会评估 Base 表、所有 Rollup 和物化视图的执行代价，如果发现查询条件能够被某个 Rollup 或物化视图满足，优化器会自动将查询路由到该索引上读取预聚合的数据，从而实现查询的“毫秒级”响应，整个过程对用户完全透明。

## 三、 Apache Doris 与 ClickHouse 的深度对比

在现代 OLAP 领域，Doris 和 ClickHouse 经常被拿来比较。它们都是追求极致性能的列式数据库，但在架构设计哲学和适用场景上有着显著的差异。

### 1. 架构设计哲学对比


| 维度          | Apache Doris                                                                          | ClickHouse                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **分布式架构**   | **真正的 MPP 架构**。FE 节点负责全局查询规划和分布式调度，BE 节点负责执行。节点间协同工作，对用户屏蔽底层分布式细节。                    | **相对松散的分布式集群**。本质上是单机数据库的集合。需要依赖 ZooKeeper 来协调分布式 DDL 和副本同步。                                                          |
| **表与集群关系**  | **全局视角**。在 FE 建表后，表自动分布在整个集群中。                                                        | **本地视角 + 分布式表**。需要在每个节点上建“本地表”（Local Table），然后再建一个“分布式表”（Distributed Table）将它们逻辑上缝合起来。                                |
| **Join 能力** | **强大且灵活**。原生支持 Broadcast Join、Shuffle Join、Colocate Join 等多种分布式 Join 策略，优化器会自动选择最优路径。 | **相对较弱**。ClickHouse 是单表查询的王者，但处理多表复杂 Join（尤其是大表 Join 大表）时，经常需要手动指定 GLOBAL JOIN，容易遇到内存瓶颈（OOM）。                         |
| **数据更新与删除** | **原生支持高频更新**。Unique Key 模型（特别是 Merge-on-Write）能够很好地支持实时 UPSERT 和 DELETE 操作。           | **弱支持更新**。采用 Mutation 机制（如 `ALTER TABLE UPDATE/DELETE`），这是一个非常重的异步操作，不适合高频的实时更新，通常只能通过 ReplacingMergeTree 引擎在查询时合并去重。 |
| **扩缩容体验**   | **平滑且自动化**。增加节点后，系统自动以 Tablet 为单位进行数据均衡，业务无感知。                                        | **极其痛苦**。扩容后新节点不会自动分担老节点的数据，需要手动编写脚本进行数据重分布（Resharding），运维成本极高。                                                       |


### 2. 优势与不足总结

#### ClickHouse 的优势（Doris 的相对不足）：

1. **单表查询性能的绝对王者**：在不需要 Join 的大宽表查询（如日志检索、指标统计）场景下，ClickHouse 将 CPU 和磁盘 IO 压榨到了物理极限，单表性能通常优于 Doris。
2. **极度丰富的函数库**：ClickHouse 提供了大量奇技淫巧的内置函数（如各种漏斗分析、数组处理、URL 解析函数），在某些特定分析场景下极为方便。
3. **生态成熟度**：ClickHouse 开源较早，在全球范围内拥有庞大的用户群和极其成熟的周边生态。

#### Doris 的优势（ClickHouse 的相对不足）：

1. **极简运维，真正的“开箱即用”**：无需依赖 ZK，建表不需要分本地表和分布式表，扩缩容全自动，对 DBA 和开发人员极其友好。
2. **强大的复杂查询与 Join 能力**：Doris 的优化器（基于 Cascades 框架）更加智能，能够像传统数据仓库一样轻松处理星型/雪花模型的多表复杂关联。
3. **完善的实时更新能力**：完美契合 MySQL Binlog 同步、订单状态变更等需要高频 UPSERT 的业务场景。
4. **标准 SQL 兼容性**：高度兼容 MySQL 协议和标准 SQL，BI 工具接入几乎零成本；而 ClickHouse 的 SQL 语法有许多自己独有的“方言”。

### 3. 实际业务应用场景建议

- **选择 ClickHouse 的场景**：
  - **日志分析与监控系统**：数据写入后极少修改（Append-Only），表结构通常是预先打平的“大宽表”。
  - **对单表查询性能有极致苛刻的要求**，且团队有较强的底层运维和调优能力。
- **选择 Apache Doris 的场景**：
  - **实时数据仓库与 BI 报表**：数据模型复杂，包含大量维度表和事实表的关联查询（星型/雪花模型）。
  - **订单/交易状态实时分析**：数据需要高频更新和删除（如同步 MySQL 的 Binlog）。
  - **希望降低运维成本**：团队规模不大，希望数据库能像 MySQL 一样简单易用，扩缩容省心。

## 四、 部署及运维体验

Doris 在设计之初就将“易用性”放在了极高的位置，这在部署和运维上体现得淋漓尽致。

### 1. 极简部署，不依赖 Hadoop 生态

- 只需要下载预编译的二进制包，修改少量配置文件（如 IP 绑定），即可启动 FE 和 BE 进程。
- 完全不需要安装 HDFS、ZooKeeper、HBase 等笨重的 Hadoop 生态组件，对中小企业非常友好。

### 2. 兼容 MySQL 协议

- Doris 的 FE 节点完全兼容 MySQL 协议。这意味着用户可以使用任何标准的 MySQL 客户端（如 Navicat、DBeaver、JDBC/ODBC 驱动）直接连接 Doris。
- 开发人员几乎没有学习成本，原有的基于 MySQL 的 BI 工具（如 Tableau、Superset、Metabase）可以无缝对接。

### 3. 在线平滑扩缩容

- **扩容**：只需在新机器上启动 BE 进程，并在 FE 中执行一条 `ALTER SYSTEM ADD BACKEND` 的 SQL 语句。Doris 会自动在后台进行 Tablet 的均衡迁移，整个过程不影响线上业务的读写。
- **缩容**：执行 `DECOMMISSION` 命令，Doris 会先将该节点上的数据副本安全地迁移到其他节点，确认无误后才会下线该节点。

### 4. 丰富的监控与诊断

- Doris 原生提供了基于 Prometheus + Grafana 的监控模板，可以直观地看到集群的 QPS、延迟、Compaction 状态、内存占用等核心指标。
- 提供了 `EXPLAIN` 和 `PROFILE` 命令，帮助 DBA 深度分析 MPP 执行计划中的算子耗时，快速定位慢查询瓶颈。

## 总结

Apache Doris 是一款将 MPP 架构发挥到极致的现代 OLAP 数据库。它通过极简的架构、向量化引擎、丰富的数据模型以及强悍的实时写入能力，成功解决了传统 MPP 数据仓库在实时性、高并发和运维复杂性上的痛点。无论是构建企业级实时数仓，还是支撑高并发的交互式分析，Doris 都是目前开源界最具竞争力的选择之一。
