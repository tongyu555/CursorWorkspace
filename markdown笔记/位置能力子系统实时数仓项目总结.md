# 项目总结：位置能力子系统实时数仓建设与维护

## 1. 项目基本信息

- **项目名称**：位置能力子系统实时数仓建设与维护
- **项目背景**：为对外公众产品提供高精度、低延迟的 4G、5G、VoLTE 实时位置数据服务。业务对 SLA 要求极高，要求从信令生成到发布给下游位置能力子系统的端到端总时长必须控制在 **10分钟以内**。
- **技术栈**：Apache Flink (DataStream & Table API), Kafka, HDFS/ViewFS, RocksDB, Scala/Java, NiFi

---

## 2. 个人角色与核心职责

- **角色**：实时数仓/大数据开发工程师
- **核心职责**：负责核心实时流处理程序（5G 实时位置数据解析、VoLTE 信令解析与工参关联）的设计、开发、调优及生产维护。主导了多端数据的实时清洗、异常废卡过滤、海量基站工参（CellKey）内存关联，以及最终异构双写（Kafka/HDFS）的研发工作。

---

## 3. 主要工作内容与技术亮点

### 3.1 架构设计与数据流转

- **数据源接入**：通过 NiFi 将源端采集（SFTP）的数据实时解析并推送到前置集群 Kafka，为实时程序提供高吞吐的源数据。
- **Flink 实时清洗与处理**：
  - **基于 Table API 与 DataStream 的混合处理**：利用 Flink SQL DDL 和 View 实现源端数据的快速条件过滤（如 `procedure_status=1`、过滤特定接口等），随后转换为 DataStream 进一步精细化处理。
  - **动态分流处理**：使用 Flink 的 `ProcessFunction` 和测输出流（SideOutput/OutputTag）机制，实现 4G 与 5G 信令数据的精准物理分流。对异常废弃数据（如长度越界、解析失败）进行隔离丢弃，避免脏数据污染下游。
- **异构双写输出**：清洗关联后的数据通过 `union` 合并，采用双 Sink 策略分别写入核心 Kafka（通过 SASL/SCRAM 加密供下游应用端消费）和前置 HDFS（按处理时间进行多级目录分区，供离线指标稽核）。

### 3.2 海量工参数据的关联设计（核心亮点）

- **痛点**：传统的 Broadcast（广播状态）流处理基站工参数据时，由于全省工参数据量庞大，导致 Flink Checkpoint 状态过大，频繁出现 Checkpoint 超时或失败。
- **解决方案**：摒弃 Broadcast 方案，采用 `RichMapFunction` 的 `open()` 方法结合 `ScheduledExecutorService` 开启单线程定时任务，每隔 N 小时自动从 HDFS 读取最新的 4G/5G 正常和纠偏工参数据（支持 Parquet/CSV），并加载到 JVM 级别的 `ConcurrentHashMap` 中进行缓存。在 `map` 阶段通过 `CellKey`（如 `enodebId_cellId`）极速关联内存数据，极大地减轻了 State Backend 的压力。

### 3.3 生产问题排查与性能调优

- **Checkpoint 效率优化**：将 Checkpoint 存储后端从 `JobManagerCheckpointStorage` 优化至基于 RocksDB 的 `FileSystemCheckpointStorage`（增量快照配置），大幅提升快照生成效率，规避 JobManager 内存溢出。
- **HDFS FileSink 滚动策略修复**：针对官方 HDFS 滚动策略（RollingPolicy）在历史文件滚动时导致 Checkpoint 报错的问题，深度定制了基于 Checkpoint 同步的滚动检测逻辑（`UDFDefCheckpointRollingPolicy`），彻底解决了文件写入 HDFS 时的状态不一致和报错故障。

## 4. 项目成果与业务价值

- **成功保障核心 SLA**：实时流程序并行度与 Kafka Partition 一一对应，数据处理无积压，完美实现了从信令产生到下游位置系统可用在 10 分钟以内的严格指标要求。
- **系统高可用与稳定性**：在保证处理海量信令数据的同时，规避了状态膨胀导致的 OOM 和 Checkpoint 失败等隐患，保障了核心对外公众产品 7*24 小时的稳定运行与输出。

## 5. 面试常见可能被问到的问题 (Q&A 准备建议)

- **Q1: 在做实时流与维度表（工参表）关联时，为什么没有使用 Flink 原生的 Broadcast 流？**
  - **准备思路**：重点回答 **状态大小与性能的权衡**。工参表数据量巨大，如果使用广播流，这部分数据会被持久化到 Checkpoint 中，导致每次 Checkpoint 的体积急剧膨胀，进而引起超时和失败。而改用定时加载至本地内存（`ConcurrentHashMap`）的方式，这部分静态维表数据就不需要进 State，大大减轻了快照压力。
- **Q2: 你提到使用 ConcurrentHashMap 缓存工参，那如何处理数据的更新和 OOM 问题？**
  - **准备思路**：回答通过 `ScheduledExecutorService` 定时刷新（如每隔数小时）从 HDFS 增量或全量替换 HashMap。关于 OOM，可以回答通过监控日志中缓存集合的大小，在规划 TaskManager 内存时预留足够的 Heap 空间；另外因为各个 TM 内的 Subtask 共享 JVM 内存，采用**静态（static）Map** 保证一个 TM 实例只加载一份工参数据，避免了内存的重复占用。
- **Q3: 遇到废卡或者异常长度的数据，你们在流里是怎么处理的？**
  - **准备思路**：讲解 Flink 的 **Side Output（侧输出流）** 机制。在 `ProcessFunction` 里做逻辑判断，如果是合规数据走主流（`out.collect`），如果是废弃数据或异常数据打上标签后发送到侧输出流（`ctx.output`）。这既能保证脏数据不影响主业务，又方便日后单独对侧输出流进行落盘分析监控。
- **Q4: 你在项目中遇到的 HDFS FileSink 滚动导致 Checkpoint 失败的具体原因是什么？怎么解决的？**
  - **准备思路**：体现你处理底层踩坑问题的能力。可以解释在旧版本中，基于大小或时间的异步滚动可能会刚好与 Checkpoint 动作发生竞争，导致正在被 Checkpoint 引用的文件被滚动重命名而找不到。解决办法是自定义了 `RollingPolicy`，将文件滚动的判断条件与 Checkpoint 周期同步挂钩，在 Checkpoint 触发前后安全地完成文件滚动。
