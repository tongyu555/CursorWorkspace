二面（组长 / 技术负责人）高频面试题 & 答题要点
一、项目深挖（必考，占比最高）
1. 挑一个你最核心的大数据 / 数仓项目，完整讲一遍？
答题框架（固定模板）
业务背景：对接什么业务（用户行为 / 交易 / 广告）、业务目标、数据量级（日增数据量、表数据量、并发）；
整体架构：数据采集→消息队列→数仓分层→应用，写明技术栈；
模型设计：主题域划分、分层设计、事实表 / 维度表设计思路；
核心工作：你负责的模块、核心指标、核心 SQL / 任务；
难点 & 问题：列举 2-3 个真实问题（数据倾斜、任务延迟、数据质量、小文件等）；
解决方案：具体落地手段、优化前后对比（耗时缩减、资源下降、准确率提升）；
落地价值：给业务带来的帮助。
加分提醒：主动贴合 OPPO 业务，举例可以往用户活跃、留存、广告转化、设备行为靠拢。
2. 项目中遇到过最棘手的问题是什么？怎么排查和解决的？
高频作答方向（二面最爱追问）
优先准备这两类真实场景：
数据倾斜
答：遇到大表 Join 热点 Key / 空值倾斜。先通过日志、任务监控定位倾斜 Key；空值就提前过滤，热点 Key 采用加盐打散，拆分任务计算后再合并；大表 Join 小表改用 MapJoin 优化，最终任务运行时长缩短 XX。
离线任务大面积延迟
答：先查依赖链路，确认上游任务是否阻塞；再看集群资源、分区数据量暴涨；排查 SQL 是否存在低效逻辑、小文件过多；通过合并小文件、优化 Shuffle、调整并行度、拆分大任务解决。
3. 你们数仓是如何做分层和主题域划分的？为什么这么设计？
答：标准四层架构 ODS/DWD/DWS/ADS。
ODS：同步原始日志、业务库数据，不做清洗，保证原始数据可回溯；
DWD：清洗脏数据、去重、补全字段，保留明细粒度；
DWS：按用户、渠道、时间等维度轻度聚合，产出中间指标，复用性最强；
ADS：面向报表、大屏、业务分析，产出最终指标。
主题域按照用户、设备、行为、广告、渠道划分，拆分后职责清晰、减少耦合、方便团队协作与指标复用。
4. 如何设计一张用户行为明细表？核心字段、分区、分桶怎么设计？
答：
核心字段：唯一用户 ID / 设备 ID、行为类型、行为时间、页面 ID、渠道、设备型号、操作系统等；
分区：按天 (dt) 为主，部分场景追加小时分区，缩小查询范围；
分桶：高频关联字段（user_id/device_id）做分桶，优化多表 Join、抽样查询效率。
二、数仓架构 & 数据治理（深度考点）
1. 什么是缓慢变化维？工作中如何处理？举实际例子
答：维度属性随时间发生变更，就是缓慢变化维，常见于用户维度、商品维度。
SCD1（直接覆盖）：只保留最新数据，适合无需追溯历史的场景，比如用户昵称修改；
SCD2（拉链表）：保留全历史版本，通过start_time/end_time标记数据生命周期，用户等级、会员状态变更常用，也是工作主流方案。
2. 你们团队如何保障数据质量？有哪些监控手段？
答：围绕完整性、准确性、唯一性、及时性四大维度落地：
完整性：校验分区数据量、关键字段空值率，低于阈值告警；
唯一性：对唯一主键做重复数据校验；
准确性：核心指标做交叉校验（多链路数据比对）；
及时性：监控任务运行时长、分区产出时间，超时告警。
落地方式：定时执行校验 SQL、接入监控平台、异常消息推送，问题第一时间修复。
3. 数据血缘、元数据在工作中起到什么作用？
答：
元数据：记录表结构、字段含义、分区、任务调度信息，统一口径，新人上手、口径核对必备；
数据血缘：追踪数据上下游链路，指标异常、任务报错时快速定位源头；变更表结构时，提前评估影响范围，避免线上事故。
4. 冷热数据如何管理？数据生命周期怎么配置？
答：热数据（近 30 天）正常存储，高频查询；冷数据（半年以上）开启压缩、归档至低成本存储；
按业务规则设置分区 TTL，自动清理超期冷数据，减少集群存储压力，一般日志类数据保留 3-6 个月，核心交易 / 广告数据保留 1 年以上。
三、Spark & Hive 核心调优（重中之重）
1. Spark Shuffle 原理？日常有哪些优化手段？
答：Shuffle 是数据跨分区重分发的过程，分为 Map 端写数据、Reduce 端拉取数据两个阶段。
优化手段：
合理设置分区数，避免分区过多产生大量小文件，或分区过少导致单个任务压力大；
开启 Shuffle 数据压缩，减少网络 IO；
优化分区字段，规避热点 Key；
调整缓冲区、拉取重试等参数，提升稳定性。
2. Spark 和 Hive 执行同一任务，性能差异在哪？什么场景优先用 Spark？
答：Spark 基于内存计算，中间数据不落盘，迭代计算、复杂 SQL、大表计算性能远优于 Hive（MapReduce）；
纯简单查询、小批量数据可使用 Hive；大数据量 ETL、复杂聚合、多轮计算场景优先使用 Spark SQL。
3. 大表 Join 除了数据倾斜，还有哪些优化方案？
答：
小表优先使用广播 Join，避免全局 Shuffle；
提前过滤无用数据、裁剪字段，减少数据传输量；
对关联字段做分桶，利用桶表 Join 优化；
拆分超大任务，分批次执行。
4. Hive 小文件过多有什么危害？怎么解决？
答：危害：占用 NameNode 元数据、任务启动耗时变长、查询 IO 效率低。
解决：
执行alter table 表名 concatenate合并分区内小文件；
调整 Reduce/Spark 任务输出分区数；
写入阶段开启文件合并策略；
规范分区粒度，避免创建过多细碎分区。
四、实时计算（Flink+Kafka 基础考点）
1. 讲讲 EventTime、Watermark 的作用，如何处理迟到数据？
答：
ProcessingTime：服务器系统时间；EventTime：数据本身携带的业务发生时间，统计结果更精准；
Watermark：用于处理乱序数据，标记 “该时间之前的数据已全部到达”，触发窗口计算；
迟到数据：设置窗口允许迟到时间，超期数据可写入侧输出流单独处理。
2. Flink Checkpoint 作用？生产环境如何配置？
答：Checkpoint 定期保存算子状态快照，集群故障时可恢复数据，保障 Exactly-Once 语义。
生产配置：根据业务延迟要求设置间隔，开启增量 Checkpoint，配置外部持久化存储，同时搭配 Savepoint 用于版本迭代、任务重启。
3. Kafka 消息积压怎么排查？
答：
查看监控面板，确认对应分区的消费延迟数值；
排查消费端：消费逻辑是否卡顿、线程数是否不足、消费组配置异常；
排查上下游：生产端流量突增、网络 IO 瓶颈；
临时方案：增加消费线程、拆分分区；长期优化消费逻辑。
五、业务场景题（OPPO 专属，高频）
1. 核心指标 DAU / 广告转化率突然下跌，完整排查流程？
标准流程（按顺序答）
核对指标口径：是否近期修改过统计逻辑、字段、时间范围；
排查数据链路：埋点采集、Kafka、同步任务是否延迟、报错、丢数；
校验数据质量：是否出现大量脏数据、空值、重复数据；
对接业务侧：APP 版本更新、活动下线、渠道投放调整、新品动作；
定位根因，修复问题并复盘。
2. 大促 / 版本上线期间，如何保障数据链路稳定？
答：
集群资源提前扩容，调高核心任务资源配额与执行优先级；
优化 SQL、Shuffle 逻辑，提前压测；
增加全链路监控与告警，专人值守；
拆分超大任务，避免单点故障；
预留兜底方案，出现问题快速回滚。
3. 埋点数据经常出现丢数、重复，怎么治理？
答：
客户端规范埋点上报逻辑，做本地缓存与重试，减少丢数；
数据接入层基于唯一标识做去重；
全链路监控埋点上报量，波动及时预警；
统一埋点协议，定期核对埋点字典，下线无效埋点。
六、综合问答（技术 + 职业）
1. 离线数仓和实时数仓在架构、开发上有什么区别？
答：
离线：Hive+Spark，T+1 跑批，分层完整，侧重全量统计、日报 / 周报，对延迟无高要求；
实时：Flink+Kafka，秒 / 分钟级延迟，分层简化，侧重实时大屏、广告实时转化、故障监控。
2. 你如何看待 OPPO / 叶子的用户行为、广告数据业务？
参考回答：OPPO 设备体量庞大，用户行为、广告场景数据量级大、业务场景成熟，数据链路和指标体系完善；我希望结合自身数仓、大数据开发经验，深耕这块业务，同时持续提升技术能力。
3. 工作中遇到需求不合理、口径反复变更，怎么处理？
答：先和产品、运营沟通，梳理业务真实诉求，指出不合理点并给出优化建议；如果口径需要变更，第一时间同步团队，更新指标文档、同步下游应用，做好版本记录，避免数据错乱。
