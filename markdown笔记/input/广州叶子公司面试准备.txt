一、SQL 高频题（必考，手写 + 优化为主）
（一）基础语法 & 窗口函数（重中之重）
窗口函数有哪些？区别是什么？举例使用场景
答题要点：
排序类：row_number()（连续序号，不跳号）、rank()（并列跳号）、dense_rank()（并列不跳号）
聚合类：sum/avg/count/max/min() over(partition by ... order by ...)
偏移类：lag/lead() 取上 / 下一行数据
场景：用户每日排名、取用户最近一条行为、连续登录统计。
如何查询连续登录 N 天的用户？（电商 / APP 行为经典题）
核心思路：分组 + 日期差值分组。
模板思路：
去重用户 + 日期；
按用户分区、日期排序，用 row_number() 生成行号；
日期 - 行号 得到分组标识，连续日期差值一致即为同一组；
统计每组天数，筛选大于 N 天的用户。
解释 partition by 和 group by 区别
group by：聚合后行变少，一组只输出一条；
partition by：分区不合并行，每行都保留，常用于行间计算、排名。
PV、UV 如何计算？UV 去重方式有哪些？
PV：count(1)；UV：count(distinct user_id)
优化：大数据量下用 Bitmap、近似去重（Hive approx_count_distinct） 降低开销。
（二）SQL 性能优化 & 数据倾斜（高频难点）
Hive 大表 Join 出现数据倾斜，现象、原因、解决方案？
现象：任务卡住、部分 Reducer 耗时极长、整体跑批慢；
常见原因：空值 / NULL 过多、热点 Key、数据分布不均；
解决方案：
过滤无效空值；
热点 Key 加盐打散、二次聚合；
小表广播 map join；
拆分倾斜 Key 单独计算再合并。
Hive 如何优化大数量级的 count(distinct)？
方案 1：先分组 group by user_id 再外层 count(1)；
方案 2：使用 Bitmap 函数；
方案 3：近似去重函数 approx_count_distinct（允许误差场景）。
Hive 分区、分桶的作用与区别？日常怎么使用？
分区：按时间 / 地区等维度拆分目录，缩小扫描范围，最常用按天分区 dt；
分桶：对列哈希拆分文件，优化 Join、抽样、查询效率；
实践：离线数仓必做分区，大表关联场景加分桶。
小文件产生原因、危害及 Hive 如何合并小文件？
危害：NameNode 压力大、查询启动开销高；
解决：alter table concatenate、调整 Reducer 数量、写入前合并、使用 CombineInputFormat。
（三）业务类 SQL 场景题（贴合 OPPO 电商 / 用户行为）
统计每日 DAU、MAU、新增用户、次日留存率
DAU：当日独立用户数；
新增：当日首次出现的用户；
次日留存：今日活跃用户，次日仍活跃的占比。
电商场景：统计每个用户最近 3 笔订单、订单金额 Top10 商品
考点：row_number() 分区排序 + 取 Top。
漏斗模型如何实现？比如：首页点击→商品详情→加购→下单→支付
思路：按用户分组，标记各行为节点，判断行为链路完整性，分层统计转化。
二、数据仓库专题（核心考察，分层、建模、规范）
（一）数仓分层理论
标准数仓分层 ODS/DWD/DWS/DM 每层作用、数据特点、使用规范？
ODS 原始数据层：同步源系统全量 / 增量数据，几乎不清洗，保留原始格式；
DWD 明细层：清洗、去重、过滤脏数据、补全字段，保留明细粒度；
DWS 汇总层：轻度聚合，按用户 / 商品 / 渠道等维度汇总，中间指标；
DM 应用层：面向业务报表、看板、专题分析，最终指标数据。
为什么不直接用 ODS 层做报表？
原始数据脏数据多、格式杂乱、重复数据多、查询效率低、无法复用，分层是为了解耦、复用、易维护、性能优化。
（二）数据建模
维度建模是什么？星型模型、雪花模型区别，工作中用哪种？
维度建模：数仓主流建模，分为事实表 + 维度表；
星型：维度表直接关联事实表，查询快、主流使用；
雪花：维度表再拆分多层，冗余低但关联多、性能差，极少用。
事实表、维度表怎么区分？举电商 / APP 行为例子
事实表：存行为、度量值（订单表、点击日志、支付记录）；
维度表：存描述属性（用户维度、商品维度、地区维度）。
（三）数据治理 & 质量
工作中如何做数据质量监控？监控哪些指标？
监控维度：数据完整性（空值）、准确性（逻辑校验）、唯一性（重复数据）、及时性（数据延迟）；
实现：定时校验 SQL、调度任务告警、监控平台、异常推送。
数据生命周期管理怎么做？冷热数据如何处理？
按分区设置 TTL 自动清理历史冷数据；
热数据：常规存储，高频查询；冷数据：归档、压缩、迁移至低成本存储。
（四）业务数仓场景（叶子科技核心：用户 + 电商）
用户行为数仓和交易数仓设计上有什么差异？
行为数仓：数据量大、明细多、实时性要求高、侧重链路 / 留存 / 画像；
交易数仓：数据精度要求极高、金额 / 订单不能出错、侧重统计、结算、ROI。
埋点数据接入数仓整体流程？
埋点上报 → 消息队列 (Kafka) → ODS 层接入 → 清洗 (DWD) → 汇总 (DWS) → 报表 / 应用。
三、大数据组件面试题（Hadoop/Spark/Flink/Kafka，必问）
（一）Hadoop & HDFS
HDFS 架构？NameNode、DataNode 作用？
主从架构：NameNode 管理元数据、目录结构；DataNode 存储真实数据块；
副本机制、读写流程简单口述即可。
Hadoop 三大组件：HDFS、YARN、MapReduce 各自职责
HDFS：分布式存储；YARN：资源调度；MR：分布式计算框架。
（二）Spark（离线计算核心，重点）
Spark 和 MapReduce 相比优势？
基于内存计算，中间结果不落盘，迭代计算更快；DAG 调度，执行效率更高；算子更丰富。
Spark RDD、DataFrame、Dataset 区别？
RDD：弱类型、无 Schema；
DataFrame：有 Schema、列结构，类型弱；
Dataset：强类型，兼顾类型安全与性能，三者都支持惰性执行。
Spark Shuffle 原理、阶段、如何优化 Shuffle？
Shuffle 是分区重排过程，分为 Map 端、Reduce 端；
优化：调整分区数、压缩、合并小 Shuffle 文件、合理使用分区字段。
Spark 运行架构：Driver、Executor、Cluster Manager 作用
Driver：主进程，解析任务、生成 DAG、调度；
Executor：从节点，执行 Task、存储数据；
资源管理器：Yarn/K8s 等。
（三）Flink（实时计算，常考基础）
Flink 相比 Spark Streaming 优势？
真正流式处理（逐行处理），低延迟；支持 EventTime、Watermark、状态管理； Exactly-Once 语义更完善。
解释 EventTime、ProcessingTime、Watermark 水印作用
ProcessingTime：服务器系统时间；EventTime：数据自带事件时间；
Watermark：处理乱序数据，标记 “小于该时间的数据不会再来”，触发窗口计算。
Flink 窗口分类？滚动窗口、滑动窗口、会话窗口区别与场景
滚动窗口：无重叠，如每小时统计；
滑动窗口：有重叠，如每 10 分钟统计近 1 小时数据；
会话窗口：按空闲间隔划分，用户行为会话分析。
Flink 状态是什么？状态持久化 & Checkpoint 作用
状态：算子中间计算结果；Checkpoint 定期保存状态，故障自动恢复，保证数据一致性。
（四）Kafka（数据接入层）
Kafka 架构：生产者、消费者、Broker、分区、副本作用
分区：并行读写、提升吞吐量；副本：高可用，主副本挂掉从副本顶上。
Kafka 消息可靠性如何保证？（生产、存储、消费三端）
生产端：ACK 机制、重试；
存储端：多副本、持久化落盘；
消费端：手动提交 offset、位移管控。
Kafka 如何处理消息重复、消息丢失？
丢失：优化 ACK、副本、offset 提交时机；
重复：业务层幂等设计（唯一主键去重）。

线上指标突然下跌 / 异常，你的排查思路？（高频压力题）
标准流程：
确认口径是否变更；
检查数据链路：采集、队列、同步任务是否延迟 / 报错；
校验数据质量：脏数据、空值、重复；
核对业务侧：版本更新、活动下线、渠道变动；
定位根因 + 修复 + 复盘。
离线跑批任务延迟，排查方向？
资源不足、数据倾斜、小文件过多、依赖任务阻塞、分区数据量暴增、SQL 性能差。
