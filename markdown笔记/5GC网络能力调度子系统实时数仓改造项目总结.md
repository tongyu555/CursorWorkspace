# 面试项目总结：5GC网络能力调度子系统 - 实时数仓批处理改造

## 1. 项目基本信息

- **项目名称**：5GC网络能力调度子系统实时数仓改造（5GC业务批处理改造与变更）
- **项目背景**：在原架构中，5G DPI、AMF/SMF 业务日志、5G CGF 话单（共15类重点数据）采用 Flink 实时消费 Kafka 并直接写入 HBase。随着 5G 业务的爆发式增长，高频的实时读写导致了 Kafka 副本同步瓶颈以及 HBase 集群负载过重（如：Region 极度不均衡、请求频率爆表）。为彻底释放集群读写压力并兼顾 SLA 保障，决定进行架构重构：将实时入库改为基于 Spark 的离线批量入库。
- **技术栈**：Spark (批处理), HDFS, HBase (BulkLoad), DS 离线调度平台, Flink (旧链路替换), NiFi

---

## 2. 个人角色与核心职责

- **角色**：大数据开发工程师
- **核心职责**：主导了整个 5GC 数据入库链路的架构重构。负责 **Spark 离线清洗与 BulkLoad 任务**的设计开发，HBase 预分区策略与 RowKey 优化设计。同时，主导了涵盖安徽等 16 个 H3 省份的生产环境分批割接、资源动态调度调整以及历史数据查询链路的兼容适配。

---

## 3. 主要工作内容与技术亮点

### 3.1 架构重构：实时改批处理与 BulkLoad 极速入库

- **原链路痛点剥离**：取消 NiFi -> Kafka -> Flink -> HBase 的高负载实时链路，重构为 NiFi -> HDFS -> Spark (清洗及生成 HFile) -> HBase BulkLoad 的“准实时”批处理链路。彻底砍掉了 Kafka 在该业务上的压力。
- **BulkLoad 极致优化**：通过 Spark 读取 HDFS 上的贴源表，进行数据清洗并直接生成 HBase 底层能够识别的 HFile 文件。然后利用 BulkLoad 机制绕过 HBase 的 WAL 和 MemStore 缓存直接将文件加载入库。这种方式将入库效率提升了数倍，彻底解决了原 Flink 直写导致 HBase 请求堆积和集群高负载的问题。

### 3.2 HBase 表结构设计与全生命周期管理

- **动态预分区与 RowKey 设计**：针对 5GC 海量日志导致的 Region 热点问题，设计了高散列性的 RowKey 结构：**分区随机数 + 反转手机号 + 时间戳 + 序列号后5位**。结合 HDFS 前一日数据量，在 Spark 程序中动态计算建表所需的预分区数（公式：`Size/60GB + 1`），使得数据在写入时天然均匀分布，消除了读写热点。
- **生命周期自动化**：在 DS 调度平台上设计工作流，开发了支持“每日动态创建新表（预分区 + 独立生命周期）”和“定时清理过期表（避免磁盘爆满）”的闭环自动化管理机制。

### 3.3 生产级割接与兼容性适配

- **无缝查询兼容**：在 16 个省份的割接过程中，存在新老 HBase 架构交替期。开发并上线了应用端查询适配服务逻辑：优先查当天天表，遇到时间跨度查询时自动拆分并组合查询天表和历史老表，确保应用端无感平滑切换。
- **大规模集群的资源治理与割接**：制定了详尽的 4 省/批次割接方案，包括停用旧版 Flink 程序、调度新版 Spark 工作流。割接后，根据各省 15 类数据量的实际差异（分级配置 8Core~30Core），对 YARN 资源队列进行了细粒度的切分与动态资源平衡，确保任务在 20 分钟内跑完。

## 4. 项目成果与业务价值

1. **集群性能显著跃升**：彻底甩掉了 Kafka 集群在 5GC 业务上的负担，HBase 的请求频率和总并发数（totalRequestCount/s）断崖式下降，集群健康度（异常 Region 数）大幅好转。
2. **零阻碍的业务割接**：高质量地完成了天津、河北、安徽等 16 省份的分批割接，未发生数据断层或下游能力系统查询异常，保障了 5G 调度能力、反诈等重点业务支撑系统的稳定。
3. **沉淀高可用离线数仓架构**：沉淀了一套“HDFS源数据量预估 -> 动态预分区建表 -> Spark 生成 HFile -> BulkLoad 导入 -> 生命周期自清理”的标准工业级海量数据入库解决方案，可复用于后续其他日志系统的开发。

## 5. 面试常见可能被问到的问题 (Q&A 准备建议)

- **Q1: 为什么把 Flink 实时写 HBase 改成 Spark 离线（批处理）写？这不会导致数据延迟变高吗，业务方能接受？**
  - **准备思路**：这是这个项目最核心的架构决策考点。首先承认延迟确实从“秒级”退到了“小时级/15分钟级”（看你们实际调度批次），但要强调这是**综合评估集群安全与资源利用率的折中**。因为海量 5GC 日志直写 HBase 会引发严重的写放大、WAL 瓶颈以及 Region Split，导致整个底座极其不稳定。与业务方对齐后，采用小批次高频调度（比如每 15/30 分钟跑一批 BulkLoad）是能满足其报表和回溯需求的，同时为公司省下了巨大的服务器扩容成本。
- **Q2: 你们使用 Spark 生成 HFile 然后 BulkLoad 进 HBase，具体原理和踩过的坑能说一下吗？**
  - **准备思路**：讲述 BulkLoad 原理：直接写到底层 HDFS 上生成 HBase 特有格式的文件，然后让 HBase 把这些文件注册进 Region。坑：HFile 内部要求数据必须**严格按照 RowKey 字典序排序**！所以在 Spark RDD/DataFrame 存成 HFile 之前，必须在 Spark 端做一个全局的 SortByKey 操作，如果这一步没做好或者产生数据倾斜，生成 HFile 就会失败。
- **Q3: 你们在建表时设计的 RowKey（分区随机数+反转手机号+时间戳），为什么要这么设计？以及怎么动态评估预分区的？**
  - **准备思路**：展现你对 HBase 底层原理的掌握。**分区随机数/散列列**是为了解决写热点；**反转手机号**是因为手机号前缀通常是同质化的（如138、139），反转后可以让尾号（变化最大）处于前列，进一步增加散列性；**时间戳**放在后面是为了应对有时序查询的需求。至于预分区，因为一个 Region 最佳容量通常是几 GB 到数十 GB 不等，所以根据前一天 HDFS 文件大小（`Size/60GB+1`）来动态建区，能保证每个 Region 数据足够饱满又不会马上撑爆发生分裂。
