# 网络数据处理系统前置31省H3迁移项目总结

## 1. 项目基本信息

- **项目名称**：企业级大数据平台网络数据处理系统
- **项目背景**：为解决现网痛点，响应集团“降本增效、自主可控”及集群安全要求，将原有的PaaS底座（H2）平滑升级并迁移至全新的自研H3底座。该系统采用“核心+前置”两级分布式架构，承载全网海量网络数据（DPI、MR、CGF、CDR等）的统一汇聚与处理，服务于国家反诈中心、5G调度能力系统等高SLA要求的核心业务。
- **技术栈**：
  - **大数据存储与计算**：Hadoop (HDFS, YARN), HBase, Hive, Spark
  - **实时计算与消息中间件**：Flink, Kafka
  - **数据集成与同步**：NiFi, DataX
  - **任务调度与监控**：DolphinScheduler (DS), ZERO平台, Prometheus, Grafana

---

## 2. 个人角色与核心职责

作为前期工作的主要牵头人及核心开发工程师，我主要负责**贴源数据的接入与适配、新版DS调度平台的功能验证、离线与实时处理任务的迁移调优、全量数据质量的校验**，以及最终**生产环境的业务平滑割接与保障**。

---

## 3. 主要工作内容与技术亮点

### 3.1 新版 DS (DolphinScheduler) 调度平台验证与应用

- **痛点与挑战**：新底座采用了全新的 DS 调度平台，需确保海量任务平滑过渡，不出现调度紊乱或资源死锁。
- **解决方案**：前期牵头对 DS 平台功能进行全面验证。测试了集群 Master/Worker 的心跳监控、资源隔离机制，并针对 HDFSCHECKER、Spark 和 DataX 任务类型的参数配置（如重试机制、执行间隔、并发 Channel 数等）进行了深度验证和批量核查，为后续几百个工作流的稳定运行奠定了基础。

### 3.2 贴源数据接入与离线任务的适配迁移

- **贴源数据接入**：主导设计并实施基于 NiFi 的数据流向变更（从旧 HDFS 路径映射至新 H3 HDFS 路径），管理 4G/5G MR、DPI、物联网日志等海量源数据的接入。
- **离线任务迁移与调优**：负责 40张贴源表、108张结果表的适配与离线计算逻辑迁移。
- **性能调优**：建立了一套离线任务资源动态调整机制。例如对执行超10分钟的 DataX 任务提升资源档位（如升至 30 channel+8G），对执行超2小时的 Spark 天级任务和超1小时的小时级任务进行计算资源扩容，显著降低了数据处理延迟。

### 3.3 实时任务的迁移测试与验证

- **场景**：涉及 4G/5G 实时位置数据、5GC调度数据等高吞吐、低时延要求的业务场景。
- **实施过程**：将原平台的 Flink 实时流程序迁移至新环境，在 ZERO 平台上重新配置参数并启动。
- **测试验证**：通过并行双跑机制，在核心 Kafka 节点消费数据进行对比验证，重点监控端到端时延（必须保障在10分钟以内）、处理吞吐量及数据格式准确性，确保了消息无积压、消费零丢失。

### 3.4 全方位数据质量验证（测试与校验）

为保证数据一致性，我主导制定了**数据质量校验规则及对比方案**：

- **一致性比对**：通过编写 SQL 对比新老集群的关键字段（例如：记录数、平均端到端时延、定位字段非空率、定位精度），通过计算关键指标波动率，确保波动率稳定在 ±5% 的安全阈值内。
- **多维度校验**：
  - **接口层面**：通过编写脚本批量验证 HBase 提供的 API 查询接口（CGF、AMF/SMF），检查接口状态码与返回内容的准确性。
  - **容量层面**：比对 HBase 库与 Hive 表的 RowCount，确保离线入库数据条数 100% 一致。
- **成果**：完成 108张结果表、42张贴源表连续3天的全量数据一致性 100% 达标评定。

### 3.5 业务割接与应急保障

- **割接方案设计与演练**：设计了包含双推并行、逐步关停老平台、数据补数在内的详细割接流程。针对向核心生产环境的数据同步，先在隔离的 `_h3test` 库进行数据试推，保障零风险。
- **平滑割接**：执行期间，利用新旧 NiFi 组件和 Flink 程序的重叠运行窗口（约3-5分钟并行期），实现了国家级反诈移动日志留存（100%完整度要求，两小时内上报）和实时位置服务等高 SLA 业务的不中断割接。
- **应急回退闭环**：制定了完善的失败回退机制（如重置配置、重启老集群组件、修正数据源 IP），为系统切换上了“双保险”。

## 4. 项目成果与业务价值

1. **降本增效与自主可控**：成功将近60台规模的前置节点集群平滑迁移至集团自主研发的H3底座，彻底解决了老平台痛点。
2. **零重大故障割接**：在不停机、不影响下游核心业务（如工信部防诈骗日志上报、电信联通共建共享生产调度云）的前提下，完成了1200+TB 日均数据量处理任务的平稳迁移。
3. **沉淀规范与资产**：梳理出了一套完整的大数据组件健康检查手册、DS调度平台避坑指南以及海量数据验证的方法论，极大地提高了团队在此类大型集群迁移项目的实施效率。

## 5. 面试常见可能被问到的问题 (Q&A 准备建议)

- **Q1: 你们在做新老平台数据校验的时候，具体是怎么做的？如何保证数据完全一致？**
  - **准备思路**：回答制定了±5%的波动率标准，利用编写批量的Hive/Spark SQL脚本去同时跑新旧两套表的数据（如计数、极值、均值），重点核对比如基站定位精度、时延等关键业务指标。并且通过脚本自动比对。
- **Q2: 离线任务（DataX/Spark）迁移过程中遇到了什么性能瓶颈，你是怎么调优的？**
  - **准备思路**：可以举例讲DataX任务同步由于底层HDFS或网络问题变慢，通过监控执行时间，把资源分成了几档（5/30/60 channel），对超出10分钟的任务进行了调档。
- **Q3: 实时任务割接时，怎么保证数据不丢不重？**
  - **准备思路**：回答采用了“新旧双跑+下游短时间并行消费处理”的策略，接受并行窗口期（约3分钟）的少量重复数据（交由下游利用消息ID去重），但绝不接受漏采。通过NiFi list扫描组件的暂停与清理动作来控制偏移量。
- **Q4: 新引入的DolphinScheduler平台，在前期验证时发现了哪些需要注意的点？**
  - **准备思路**：可以谈谈Worker节点的资源隔离、Master的健康检查机制，以及你们是如何通过跑批量 SQL 查询 DS 底层 MySQL 库来批量检查所有工作流配置参数（如 failRetryTimes, timeout）是否有遗漏的。
