# 中国电信车联网运营支撑平台 - 数仓离线建模与开发

## 1. 项目基本信息

- **项目名称**：中国电信车联网运营支撑平台
- **项目背景**：车联网业务运营人员普遍具备业务诉求理解能力，但缺乏深入的通信网络排障经验。本项目旨在通过接入海量车联网通信数据，利用轨迹识别和感知打标等核心算法，精准还原车辆用户发生问题时的网络状况、行驶轨迹及感知现状。通过可视化的应用大屏和诊断视图，辅助运营人员快速定位问题地点及根因。
- **核心数据**：4G/5G MRO（测量报告）详单、4G/5G DPI（深度包检测）详单、全国路网与POI数据。
- **产品目标**：实现高精度轨迹还原、用户感知概览与深度回溯，远期支持全国路网客户级覆盖评估与智驾路径规划。

---

## 2. 个人角色与核心职责

- **角色**：大数据/数仓开发工程师（离线数据清洗与建模）
- **核心职责**：在项目中负责**数仓中间层（DWD/DWM）及应用层（DWS/ADS）**离线清洗与分析任务的开发、测试验证及生产上线。共计主导完成了 **40+ 中间层模型** 和 **20+ 应用层模型** 的落地，为上层的高精度轨迹还原、路段感知打分、端到端问题回溯等核心业务提供了稳定可靠的数据底座。

---

## 3. 主要工作内容与技术亮点

### 3.1 多源海量信令数据的解析与空间拟合（中间层开发 40+ 模型）

- **数据标准化与去噪**：针对每天产生的天量 4G/5G MR 与 DPI 原始黑盒详单，利用大数据计算引擎（Spark/Hive SQL）进行数据解析、空值过滤与异常漂移点去噪，将其转换为规整的明细宽表。
- **空间轨迹拟合与关联**：将海量 MR 采样点（包含 RSRP 信号强度等质量指标）的经纬度数据与全国路网数据、基站工参图层数据进行空间映射关联（MR点拟合至路网）。为前端“出行路段网络覆盖率统计”、“轨迹 GIS 可视化”等需求提供了坚实的空间中间表支撑。

### 3.2 业务感知打标与多维特征清洗（应用层开发 20+ 模型）

- **构建端到端感知事实宽表**：基于清洗后的 DPI 数据，深度提取车辆用户在网页浏览、视频播放、即时通信、游戏等具体业务场景下的关键特征（如：HTTP状态码、DNS响应时延、视频首屏时延/卡顿频率、TCP重传率等），构建多维度的用户感知事实回溯表。
- **数据聚合与业务感知诊断**：严格按照业务方定义的感知打分与定界规则，对清洗后的感知数据按“时间-用户号卡-出行路段-APP应用”维度进行聚合计算。直接支撑上层应用的“感知概览”、“应用感知TOP排名”、“终端/无线/核心网/SP问题定界”等核心卡片功能。

### 3.3 离线计算任务性能优化与上线保障

- **海量数据的 SQL 性能调优**：针对车联网详单数据量大且基站分布不均带来的数据倾斜问题，在 Spark/Hive 离线处理环节进行了深度调优。如合理设置分区与分桶键、优化 Join 策略（如利用 Broadcast Hash Join 或加盐处理）、调整并行度等，确保了批处理任务的 T+1 稳定、按时产出。
- **高标准的数据质量验证**：与产品和算法团队紧密配合，核对各项精细化性能指标。从复杂的代码逻辑中排查问题，保障了下线交付的数据集与最终 UI 展现的数据（如：各路段弱覆盖率、网络类型驻留比、应用评分）达到 100% 的准确契合。

## 4. 项目成果与业务价值

1. **彻底打通业务侧的“网络黑盒”**：通过 60+ 个高质量的数仓模型，成功将底层晦涩的通信信令数据转化为清晰可视的业务指标，完美支撑了运营平台“高精度轨迹还原”与“用户感知诊断回溯”的核心目标。帮助车联网运营团队实现了对网络问题的“秒级查询与精确定界”。
2. **沉淀高价值的车联网数据资产体系**：规范化、分层化地落地了从原始 MR/DPI 数据到“车辆-路网-应用”感知特征的离线数仓体系。该底层架构的高扩展性，也为后续实现“全国路网客户级覆盖评估”和“智能驾驶路径规划”奠定了坚实的数据基础。

## 5. 面试常见可能被问到的问题 (Q&A 准备建议)

- **Q1: MR 和 DPI 详单数据量极其庞大，在做离线清洗和多表 Join 时遇到过什么性能瓶颈？你是如何优化的？**
  - **准备思路**：重点准备**“数据倾斜”**的实战经验。例如：在将 MR 数据与基站工参做 Join 时，因为某些核心路段基站（热点基站）的信号点极多，导致个别 Task 运行极慢卡死。可以讲讲你是如何通过过滤空值/异常值、提取热点 Key 进行单独加盐（Salting）Join，或者对于小表工参直接使用 Broadcast Join 来大幅提升跑批效率的。
- **Q2: 需求里提到了将 MR 点拟合到路网并关联工参，你们在数仓里是怎么做这种“空间位置数据”的匹配的？**
  - **准备思路**：解释空间计算的优化策略。如果是直接拿经纬度去跟地图多边形算相交会非常耗时。可以提一下你们是否用到了空间网格编码（如 Geohash 或 Uber 的 H3 算法），先把经纬度转化为网格 ID，然后再进行等值匹配（Join），极大提升了轨迹点与路网路段关联的计算速度。
- **Q3: 需求中有很多“定界诊断（比如判断是终端还是网络问题）”和“感知打分”，在应用层（ADS）建模时这块是怎么落地的？**
  - **准备思路**：向面试官展现你的**业务理解能力**。说明你是如何基于底层的具体协议指标（例如：DNS查询失败率高归类为网络或SP问题，TCP频繁重传伴随RSRP信号差归为弱覆盖问题）利用 CASE WHEN 等逻辑进行特征工程提取的，最终聚合成宽表输出，满足前端平台对“业务概览”及“回溯明细”的快速查询。
