# 面试项目总结：企业级大数据平台网络数据处理系统 - 智能中台H3迁移

## 1. 项目基本信息
- **项目名称**：企业级大数据平台网络数据处理系统（智能中台H3迁移及XC信创替换）
- **项目背景**：原有网络数据处理系统技术栈老旧且对国外商业组件依赖度高，无法满足国家信创（XC）安全自主可控要求。为解决此问题，需将涵盖全国31省份的现网庞大业务全面平滑迁移至“网络新中台（H3平台）”，实现从基础设施、数据库到大数据计算组件的全面国产化替换，同时必须保障业务连续性和数据高一致性（差异率<1%）。
- **技术栈**：
  - **大数据存储与计算**：Spark, Flink, Hadoop
  - **数据库**：TelePG / TeleDB (替换 PostgreSQL/MySQL), Apache Doris (替换 ClickHouse)
  - **数据集成与同步**：Seatunnel, NiFi
  - **日志与监控**：ELK 集群
  - **管理与服务**：智能中台低代码配置平台、安全营运台账、日志审计等

---

## 2. 个人角色与核心职责
在这个全国性的信创迁移项目中，我作为核心的**大数据开发/数据清洗工程师**，全周期主导并参与了数据采集接入、任务适配迁移及质量验证等关键工作。主要职责包括：**牵头 H3 智能中台低代码功能验证、主导源数据采集与数百项离线/实时任务的全面适配迁移、设计实施万级别核心表的数据质量稽核与一致性验证，以及支撑整体业务的双推并行与最终割接。**

---

## 3. 主要工作内容与技术亮点

### 3.1 牵头智能中台 H3 底座功能验证与评估
- **痛点与挑战**：新引进的中台底座在初期难以完全匹配复杂多样的现网业务场景，存在兼容性和易用性问题。
- **解决方案**：作为业务与底座之间的桥梁，我牵头主力进行了 H3 智能中台的低代码功能验证。结合真实业务场景深入挖掘平台瓶颈，对中台的调度功能、开发组件等提出优化建议，通过业务反推底座功能的迭代升级，避免了后期的被动适配。

### 3.2 海量源数据采集接入与同步
- **数据接入链路搭建**：负责全国31省份现网模型的梳理与标准化。配置并调试了海量的 Seatunnel 任务与多套 NiFi 采集通道，对接各类底层信令及网管数据，确保高吞吐数据接入的准确性。
- **数据库与历史数据迁移**：协同完成原关系型/元数据向 TelePG 迁移，以及核心分析型业务数据从 ClickHouse 向国产 MPP 数据库 Apache Doris 的平滑迁移。

### 3.3 离线与实时任务的全面适配迁移
- **离线任务改造**：全面梳理现网 **600+ 项离线 SQL 任务**，借助中台低代码工具将传统手工脚本转化为低代码配置任务，重新完成调度逻辑与血缘绑定，极大缩短了改造周期。
- **实时业务重构**：针对核心的 VoLTE SIP 及 4G/5G 等海量信令实时处理业务，利用 Flink（JAR包与SQL结合）进行代码重构与调度适配，特别针对防诈骗业务进行深度改造，并在中台容器环境中完成严密测试。

### 3.4 强管控的数据双推并行与质量稽核
- **双推并行验证机制**：为防范直接割接带来的数据断流风险，设计实施了“老核心+新中台”双推并行运营方案。
- **万级表数据校验**：主导数据质量一致性验证工作，设计和实现了针对 **10000+ 关键表**的数据指标稽核校验逻辑，并在跨域网络协同异常复杂的环境下，严守**“指标差异率低于1%”**的硬性验收红线，精准定位并修复数据差异。

## 4. 项目成果与业务价值
1. **全面实现信创自主可控**：系统彻底摆脱对国外基础软件的依赖，在全面国产化环境（Doris, TelePG, Flink等）下实现了同等甚至更优的数据吞吐和处理性能。
2. **海量业务零断层平稳割接**：克服了31省模型适配及跨团队网络协同难题，分批次顺利完成非建维优业务及数据共享下游的正式割接。停止老核心数据下发，达成新中台全面上线目标。
3. **沉淀标准验证体系**：输出了一套标准化的稽核算法和测试用例体系，结合模块化低代码的改造策略，不仅保障了本次迁移的质量，也为未来类似的大规模系统迁移提供了规范范本。

## 5. 面试常见可能被问到的问题 (Q&A 准备建议)

- **Q1: 在做“老核心+新中台”双推并行时，你们是如何验证数据一致性的？面对上万张表怎么保证效率？**
  - **准备思路**：回答设定了“差异率低于1%”的验收红线。提到你主导设计了自动化的稽核校验逻辑（编写标准化比对脚本/算法），通过双跑并行自动抓取两边产出的核心业务指标，快速发现差异点并逐一排查解决。
- **Q2: 在迁移 Flink 实时任务（如VoLTE或反诈业务）到新中台时，遇到了哪些挑战，怎么解决的？**
  - **准备思路**：可以讲讲 Flink JAR 与 Flink SQL 混合任务在多省份环境调度时的配置映射问题，以及如何通过规范化开发标准、在中台容器环境中反复调试来解决状态恢复或数据倾斜等难点。
- **Q3: 面对 600+ 项离线任务的迁移，你们是怎么在有限时间内完成的？**
  - **准备思路**：强调“模块化”和“低代码化”策略。借助于新中台提供的低代码配置工具，将传统的手工脚本抽象成配置项，同时重新梳理并绑定调度血缘，大幅缩短了开发周期并降低了出错率。
- **Q4: 新底座（H3中台）刚引入时不太成熟，你们是如何应对这种“边造航母边打仗”的情况的？**
  - **准备思路**：体现你的“业务前置”思维。讲讲你作为牵头人，如何深入业务场景去初步跑通闭环来验证中台底座功能，在发现如调度瓶颈、组件兼容性等痛点后，梳理成明确的需求反馈给研发团队，推动底座快速迭代优化。
