# PostgreSQL 面试重点学习笔记

> 面向面试的 PostgreSQL 核心知识梳理：架构、特性、对比、八股与常考问答。

---

## 一、架构设计

### 1.1 整体架构（多进程模型）

PostgreSQL 采用 **多进程架构**（与 MySQL 的多线程不同），每个客户端连接对应一个 **backend 进程**。

```
┌─────────────────────────────────────────────────────────────┐
│                     Client Applications                      │
└───────────────────────────┬─────────────────────────────────┘
                             │
┌───────────────────────────▼─────────────────────────────────┐
│                  Postmaster (主进程 / 守护进程)               │
│  • 监听端口、接受连接  •  fork backend  • 管理共享内存与子进程   │
└──┬──────────────┬──────────────┬──────────────┬──────────────┘
   │              │              │              │
   ▼              ▼              ▼              ▼
 Backend 1    Backend 2    Backend 3   ...   (每个连接一个进程)
   │              │              │
   └──────────────┴──────────────┴──────────────┐
                                                ▼
┌─────────────────────────────────────────────────────────────┐
│  共享内存: shared_buffers, WAL buffers, lock table, ...      │
└─────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  后台辅助进程:                                                │
│  • bgwriter   - 脏页刷盘    • checkpointer - 检查点           │
│  • walwriter  - WAL 写      • autovacuum   - 自动 vacuum       │
│  • stats collector - 统计  • logical replication - 逻辑复制   │
└─────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  存储: 数据文件、WAL、配置文件、日志等                         │
└─────────────────────────────────────────────────────────────┘
```

- **Postmaster**：负责监听、fork backend、分配共享内存、协调检查点等。
- **Backend 进程**：每个连接一个，执行 SQL、访问共享缓冲与磁盘。
- **共享内存**：`shared_buffers`（数据页缓存）、WAL 缓冲区、锁表等，进程间共享。
- **多进程带来的特点**：单连接崩溃不影响其他连接；多核扩展性好；但高并发下进程数多、上下文切换成本需考虑（连接池常用 PgBouncer 等）。

### 1.2 存储与内存结构


| 概念                        | 说明                                    |
| ------------------------- | ------------------------------------- |
| **shared_buffers**        | 数据页缓存，建议物理内存的 25% 左右，是缓存热数据的主战场       |
| **work_mem**              | 排序、hash join 等操作使用的会话级内存，按操作分配        |
| **maintenance_work_mem**  | VACUUM、CREATE INDEX 等维护操作使用的内存        |
| **WAL (Write-Ahead Log)** | 先写 WAL 再改数据页，保证持久性与崩溃恢复               |
| **Checkpoint**            | 将脏页刷盘、推进 WAL 可回收点，由 checkpointer 进程执行 |


### 1.3 事务与 MVCC

- **MVCC（多版本并发控制）**：读不阻塞写、写不阻塞读，通过元组多版本 + 可见性判断实现。
- 每行有 **xmin / xmax**（或对应 CID）标识创建/删除该版本的事务，通过当前事务 ID 与快照判断是否可见。
- **快照隔离**：默认 READ COMMITTED，可设为 REPEATABLE READ、SERIALIZABLE。
- **VACUUM**：回收死元组空间、更新可见性信息、冻结旧事务 ID，避免 wraparound；**autovacuum** 自动执行。

## 二、特点与优势

### 2.1 核心特点

- **标准兼容好**：SQL 标准遵循度高，窗口函数、CTE、递归、MERGE 等支持完善。
- **扩展性强**：插件机制（如 PostGIS、pg_stat_statements、Citus 分布式）、自定义类型与函数、多种存储引擎思路（表访问方法）。
- **功能丰富**：JSON/JSONB、数组、全文检索、分区表、逻辑复制、并行查询等。
- **可靠性高**：WAL、同步/异步复制、PITR（基于 WAL 的时间点恢复）、checksum 可选。
- **开源且协议友好**：类 BSD 的 PostgreSQL License，可商用、可二次开发。

### 2.2 与 MySQL 的常见对比（面试高频）


| 维度     | PostgreSQL              | MySQL                   |
| ------ | ----------------------- | ----------------------- |
| 进程模型   | 多进程，一连接一进程              | 多线程，一连接一线程              |
| 存储引擎   | 内置为主，可扩展表 AM            | 多引擎（InnoDB、MyISAM 等）    |
| SQL 标准 | 遵循度高                    | 历史包袱多，部分语法扩展            |
| 复杂查询   | 优化器强，复杂 SQL、多表 join 表现好 | 简单查询与主键查询优化好            |
| 并发控制   | MVCC 为主，少锁              | InnoDB 也 MVCC，锁与隔离级别需注意 |
| 复制     | 物理复制 + 逻辑复制             | 主从、组复制、binlog 复制        |
| 适用场景   | 分析、复杂业务、GIS、高一致要求       | 高并发简单读写、互联网 OLTP        |

## 三、应用场景

- **OLTP**：高一致性、复杂约束与触发器、强 ACID 的业务库。
- **OLAP / 分析**：复杂查询、窗口函数、CTE、并行查询；可配合 Citus、Greenplum 等做分布式分析。
- **GIS**：PostGIS 扩展，空间数据与地理信息。
- **全文检索**：内置 tsvector/tsquery，可与 Elasticsearch 配合或替代简单全文需求。
- **JSON 存储与查询**：JSONB 索引与 GIN，半结构化数据。
- **时序 / 监控**：TimescaleDB 等基于 PG 的扩展。

## 四、与其他关系型数据库的对比（面试常考）

### 4.1 PostgreSQL vs MySQL

- **架构**：PG 多进程 vs MySQL 多线程；PG 单引擎为主 vs MySQL 多引擎。
- **标准与语法**：PG 更贴近 SQL 标准；MySQL 有大量方言和“宽松”行为。
- **复杂查询与优化器**：PG 对多表 join、子查询、窗口等优化更成熟；MySQL 在简单主键/索引查询上延迟低。
- **复制与高可用**：PG 物理流复制 + 逻辑复制；MySQL 主从 + binlog、组复制等，生态更庞杂。

### 4.2 PostgreSQL vs Oracle

- **许可与成本**：PG 开源免费；Oracle 商业授权贵。
- **功能**：Oracle 在 RAC、高级高可用、企业级工具链上更完善；PG 在标准 SQL、扩展生态、云原生上更灵活。
- **迁移**：从 Oracle 迁 PG 需注意数据类型、PL/SQL 转 PL/pgSQL、序列与触发器差异。

### 4.3 PostgreSQL vs SQL Server

- **平台**：SQL Server 以 Windows 为主；PG 跨平台。
- **生态**：SQL Server 与 .NET、Windows 集成深；PG 与 Linux、容器、K8s 结合多。
- **高级特性**：两者都有窗口函数、CTE、分区等；PG 的 JSONB、扩展机制更开放。

## 五、重点八股文（必须掌握）

### 5.1 事务与隔离级别

- **读未提交（Read Uncommitted）**：PG 实际不会出现脏读，实现上等同于 Read Committed。
- **读已提交（Read Committed）**：每次语句开始时取快照，同一事务内两次读可能看到不同提交。
- **可重复读（Repeatable Read）**：事务开始时取快照，同一事务内读一致；PG 通过快照避免幻读（与标准可重复读略有差异）。
- **串行化（Serializable）**：通过 SSI（可串行化快照隔离）检测写冲突，保证可串行化。

### 5.2 锁

- **表级锁**：ACCESS SHARE（读）、ROW SHARE、ROW EXCLUSIVE、SHARE、SHARE ROW EXCLUSIVE、EXCLUSIVE、ACCESS EXCLUSIVE（DDL、VACUUM FULL 等）。
- **行级锁**：FOR UPDATE、FOR NO KEY UPDATE、FOR SHARE、FOR KEY SHARE；与 MVCC 配合，读多不阻塞。
- **死锁**：PG 会检测并回滚其中一个事务，应用需能重试。

### 5.3 索引类型


| 类型      | 说明                  | 典型场景                |
| ------- | ------------------- | ------------------- |
| B-tree  | 默认，等值、范围、排序         | 主键、唯一、一般查询          |
| Hash    | 等值查询（PG 10+ 支持 WAL） | 简单等值                |
| GIN     | 多值/数组/全文            | JSONB、数组、全文检索       |
| GiST    | 通用搜索树               | 几何、范围、全文（可替代部分 GIN） |
| BRIN    | 块级摘要                | 大表、按物理/逻辑顺序的范围扫描    |
| SP-GiST | 空间分区                | 非平衡结构、前缀匹配等         |


### 5.4 WAL 与恢复

- **WAL**：先写 WAL 再改数据页；崩溃后用 WAL 重放恢复。
- **checkpoint**：把脏页刷盘，并推进“可回收 WAL”的位置。
- **PITR**：基于基础备份 + WAL 归档，恢复到任意时间点。
- **同步/异步复制**：`synchronous_commit`、`synchronous_standby_names` 控制是否等备库落盘再返回。

### 5.5 复制与高可用

- **流复制（Streaming Replication）**：主库通过 WAL 流式发给备库，备库持续重放；支持同步/异步。
- **逻辑复制**：基于 WAL 解码的逻辑变更（INSERT/UPDATE/DELETE），可跨版本、选表、做异构同步。
- **高可用**：主备 + 自动故障转移（Patroni、repmgr、PG 自身无主选主，需外部工具）。

### 5.6 分区表

- **类型**：范围（RANGE）、列表（LIST）、哈希（HASH）。
- **实现**：父表为逻辑表，子表为分区；路由由约束决定，可配合分区裁剪优化。
- **PG 11+**：支持分区表上的外键、唯一约束、索引等改进。

### 5.7 关键参数（面试常问）


| 参数                                                | 含义             | 调优注意          |
| ------------------------------------------------- | -------------- | ------------- |
| shared_buffers                                    | 共享缓冲大小         | 常设物理内存 25% 左右 |
| work_mem                                          | 排序/hash 单次操作内存 | 过大导致内存占用高，按需调 |
| maintenance_work_mem                              | VACUUM/建索引等    | 加速维护操作        |
| max_connections                                   | 最大连接数          | 配合连接池使用，避免过大  |
| checkpoint_timeout / checkpoint_completion_target | 检查点频率与平滑度      | 影响 I/O 峰值     |
| random_page_cost                                  | 随机 I/O 成本      | SSD 可调低，利于走索引 |
| effective_io_concurrency                          | 异步 I/O 并发度     | SSD 可提高       |

## 六、面试常见问题与参考答案

### Q1：PostgreSQL 和 MySQL 在架构上有什么本质区别？

**答**：PostgreSQL 是 **多进程** 架构，每个客户端连接由一个 **backend 进程** 处理，进程间通过共享内存通信；MySQL（InnoDB）常用 **多线程** 模型，一个进程内多线程处理连接。多进程的好处是单连接崩溃不会拖垮整个实例，且多核利用更自然；代价是连接数很高时进程数多，需要连接池（如 PgBouncer）控制。

### Q2：什么是 MVCC？PostgreSQL 如何实现？

**答**：MVCC（多版本并发控制）通过保留数据的多个版本来实现读不阻塞写、写不阻塞读。PostgreSQL 在每个元组上记录创建/删除该版本的事务 ID（xmin/xmax 等），查询时根据当前事务的快照判断哪些版本可见，从而避免读加锁，提高并发。旧版本由 VACUUM 回收。

### Q3：为什么需要 VACUUM？和 MySQL 的 purge 有什么区别？

**答**：VACUUM 用于回收死元组占用的空间、更新可见性信息、冻结旧事务 ID（防止 32 位事务 ID 回卷）。PG 的 MVCC 在更新时产生新版本，旧版本不会立刻删除，需要 VACUUM 清理。MySQL InnoDB 的 purge 线程也是清理旧版本，但 InnoDB 的 undo 与 redo 设计和 PG 不同，PG 的 VACUUM 更“显式”，可配置 autovacuum 或手动执行。

### Q4：WAL 是什么？为什么先写 WAL 再写数据页？

**答**：WAL（Write-Ahead Log）是预写日志。修改先写入 WAL 并落盘，再在内存中改数据页，脏页可以延迟刷盘。这样崩溃恢复时只需重放 WAL 即可恢复已提交的修改，保证 **持久性（D）**。先写 WAL 再写数据，是数据库实现 ACID 的通用做法。

### Q5：PostgreSQL 有哪些常用索引类型？分别用在什么场景？

**答**：B-tree（默认，等值/范围/排序）、Hash（等值）、GIN（JSONB、数组、全文）、GiST（几何、范围、全文）、BRIN（大表顺序数据）、SP-GiST（空间分区结构）。面试时结合场景说：如 JSONB 用 GIN，大表按时间范围用 BRIN。

### Q6：流复制和逻辑复制有什么区别？

**答**：**流复制**是物理复制，以 WAL 或块为单位同步，主备数据文件一致，通常用于高可用和读写分离的只读备库。**逻辑复制**是逻辑层的变更（行级 INSERT/UPDATE/DELETE），可以只复制部分表、跨大版本、做异构同步或数据汇总，但不能用于替代物理备库做故障转移。

### Q7：如何做 PostgreSQL 的高可用？用过什么方案？

**答**：常见方案是 **主从流复制 + 自动故障转移**：用 Patroni + etcd/ZooKeeper、或 repmgr 等做主选主、VIP/ DNS 切换；应用通过连接池或读写分离访问。可补充：同步复制保证不丢已提交数据、PITR 做时间点恢复、逻辑复制做跨机房或报表库同步等。

### Q8：shared_buffers 和 OS 的 page cache 会重复缓存吗？如何理解？

**答**：会。数据先被读入 shared_buffers，刷出时写到 OS 的文件系统，可能再被 OS page cache 缓存，因此同一份数据可能既在 shared_buffers 又在 page cache，存在“双缓存”。一般 PG 建议 shared_buffers 占物理内存一定比例（如 25%），其余留给 OS 和其他 work_mem 等，避免把内存全给 PG 导致 OS 缓存不足。

### Q9：遇到过慢 SQL 如何排查？

**答**：开启 **pg_stat_statements** 看总耗时、调用次数；用 **EXPLAIN (ANALYZE, BUFFERS)** 看执行计划、实际行数、缓冲命中；查 **pg_stat_user_tables** 看表扫描与索引使用；结合 **pg_locks** 看是否锁等待。根据结果加索引、改 SQL、调 work_mem 或调 random_page_cost 等。

### Q10：PostgreSQL 的隔离级别和 MySQL 有什么不同？

**答**：PG 没有真正的“读未提交”，实现上会把读未提交当成读已提交。PG 的可重复读通过快照实现，能避免幻读；MySQL 的 RR 在部分场景下通过 gap lock 防幻读。两者都支持读已提交、可重复读、串行化，但实现细节和默认行为有差异，迁移或混用时要特别注意。

## 七、小结

- **架构**：多进程、共享内存、WAL、MVCC、VACUUM 是 PG 的核心骨架。
- **优势**：标准兼容好、复杂查询强、扩展丰富、可靠性与生态适合企业级和云原生。
- **对比**：与 MySQL 比重在进程模型、优化器与场景；与 Oracle/SQL Server 比重在开源、成本与扩展。
- **八股**：事务与隔离级别、锁、索引类型、WAL、复制、分区、关键参数要能说清。
- **面试**：结合项目说明你用过 PG 的哪些特性（复制、分区、JSONB、PostGIS 等），并准备好“为什么选 PG”“和 MySQL 如何选型”等开放题。

把上述架构、对比、八股和十问答熟记并用自己的话能讲清楚，即可覆盖大部分 PostgreSQL 相关面试场景。
