## 1. 自我介绍、项目介绍

**答题要点：**

- 控制在 1～2 分钟，突出与大数据开发相关的经历。
- 结构：教育/工作年限 → 负责的系统或数仓（分层、技术栈）→ 一两个有代表性的成果（如指标治理、任务优化、稳定性提升）。
- 项目介绍可套用：**业务背景 → 数据流（ODS→DWD→DWS/ADS）→ 技术选型（Hive/Flink/Kafka 等）→ 你负责的模块与难点 → 效果（延迟、准确率、回溯能力等）**。

**示例话术（可替换成自己的项目）：**

> 我做过 X 年大数据开发，主要负责 XX 数仓/实时链路。数仓采用 ODS、DWD、DWM、DWS、ADS 分层，离线以 Hive 为主，实时用 Flink，消息队列用 Kafka。我负责 DWS 层指标建设与任务优化，把核心报表任务耗时从 X 小时压到 X 小时，并参与数据质量与血缘建设。

## 2. 详细讲一下 DWM、DWT、DWS 这些层的具体职责和它们之间的数据流向

**参考答案：**

- **DWM（Data Warehouse Middle，中间层/轻度汇总层）**
  - **职责**：在 DWD 明细基础上做**主题域内的轻度聚合**，如按用户/设备/商品做一天或多天的汇总（如用户日行为汇总、商品日销量），为 DWS 提供可复用的中间宽表或汇总表。
  - **特点**：仍偏明细或按日/按主题汇总，不直接面向报表，减少 DWS 重复计算。

- **DWT（Data Warehouse Topic，主题宽表/主题层）**
  - **职责**：在 DWM 或 DWD 基础上，按**主题**（如用户、商品、渠道）做**拉链、全量快照或长期汇总**，形成“主题一张表”，方便做多维度分析、标签和 DWS 指标计算。
  - **特点**：偏“主题域模型”，可能包含拉链、历史快照，供 DWS 和 ADS 复用。

- **DWS（Data Warehouse Summary，汇总层）**
  - **职责**：在 DWM/DWT 或部分 DWD 上，按**统计周期（日/周/月）和统计维度（部门、渠道、产品等）** 做**指标汇总**，直接支撑报表、大屏和即席查询。
  - **特点**：以指标、维度、周期为粒度，是“可对外交付的汇总结果”。

**数据流向：**

```text
DWD（明细） → DWM（轻度汇总/主题内聚合） → DWT（主题宽表/拉链）
                    ↓                              ↓
                    └──────────→ DWS（多维度指标汇总）→ ADS/报表
```

- 简单模型下也可以是：DWD → DWS，或 DWD → DWM → DWS；DWT 是否单独一层取决于是否有“主题宽表/拉链”需求。

## 3. 如果 DWS 层某个指标突然不准了，你会怎么排查？平常怎么做血缘追踪？用的哪套工具？

**排查思路：**

1. **确认口径与取数范围**：和业务对齐指标定义、统计周期、维度（如是否去重、是否过滤测试数据）。
2. **按层反推**：从 DWS 表反查依赖的 DWM/DWT/DWD 表及分区，看对应分区的数据是否正常更新、是否有重复或缺失。
3. **对比与抽样**：用历史正确结果或业务系统数据做对比；对异常周期做抽样明细核对（如订单数、金额汇总）。
4. **任务与调度**：看依赖任务是否失败、重跑、延迟导致分区未生成或重复计算；看调度顺序是否导致用了错误分区。
5. **数据质量规则**：是否有空值、异常值、重复主键导致汇总偏差；对关键字段做非空、唯一性、数值范围校验。

**血缘追踪：**

- **做法**：在开发流程中采集“表→表”“任务→表”的依赖（建表语句、INSERT 来源、任务 SQL），形成血缘图；关键指标在文档或元数据中标注“来源表 + 字段 + 计算逻辑”。
- **工具**：常见有 **DataWorks 数据地图、Atlas、Datahub、内部数仓平台的血缘模块** 等；可回答“我们用的是 XX 平台的血缘 + 文档标注核心指标”。

## 4. 举个具体业务场景，说一下是怎么解耦的，带来了哪些好处？

**参考答案（示例）：**

- **场景**：订单域和用户域原先在同一个宽表里产出，业务经常改订单口径，导致用户侧报表也要跟着发版。
- **解耦方式**：
  - **分层解耦**：订单明细放在 DWD 订单主题，用户属性放在 DWT 用户主题；DWS 层按“订单日汇总”“用户日活跃”分别建表，报表只依赖 DWS，不直接依赖 DWD。
  - **主题解耦**：按订单、用户、流量等建主题域，各主题在 DWM/DWT 内闭环，通过**共享维度（如 user_id、date）** 在 DWS 或 ADS 再关联，避免大宽表一把梭。
- **好处**：口径变更只改对应主题或层的任务，不影响其他域；复用 DWM/DWT 可减少重复计算；故障和回溯范围可控。

## 5. ODS 到 DWS 的过程中如何保证数据质量？

**参考答案：**

- **ODS 层**：接入校验（条数、空表、分区是否到位）；对关键字段做非空、格式、枚举值校验；异常数据落隔离表或告警，不直接进入 DWD。
- **DWD 层**：去重（主键/业务键）；脏数据过滤（非法枚举、异常值）；统一编码与标准（如性别、渠道）；可打数据质量标签。
- **DWM/DWT**：对上游输入做一致性校验（如汇总与明细对账）；拉链或快照的生效日期、主键唯一性校验。
- **DWS 层**：指标结果与历史或业务系统对账；关键指标波动监控（日环比、周同比）；空值、负值、异常大值规则。
- **流程与工具**：在调度中嵌入质量规则（如 Great Expectations、DataWorks 质量）；质量不合格则阻断发布或告警；重要任务双跑或抽样核对。

## 6. 在 Hive 上处理过什么级别的数据量？任务最长耗时是多少？怎么优化的？

**答题要点（按自己经历填）：**

- **数据量**：如“单表日增 XX 亿行”“单任务扫描 XX TB”“大表 XX 亿行”。
- **最长耗时**：如“某报表任务曾跑 X 小时，优化后压到 X 小时。”
- **优化手段**：
  - 分区、分桶，裁剪扫描量；避免无效的全表扫描。
  - 小表用 map join，大表用 skew join 或加随机数打散倾斜 key。
  - 合理设置并行度（map/reduce 数）、内存、压缩；复杂 SQL 拆成多段或物化中间表。
  - 冷热分离、历史数据归档，减少常跑任务的数据量。

## 7. 如果 Hive 一个任务执行特别慢，你会怎么排查？

**参考答案：**

1. **看执行计划**：`explain` / `explain extended` 看 stage 划分、join 顺序、是否用分区/索引。
2. **看运行日志与 counters**：map/reduce 数量、数据倾斜（某些 key 的 record 数远大于其他）、spill、GC。
3. **定位瓶颈**：是某几个 reduce 特别慢（倾斜）还是 map 慢、还是单 stage 数据量过大；看各 stage 的输入输出数据量。
4. **针对性优化**：倾斜 → 加随机数打散、skew join、拆 SQL；数据量过大 → 增加分区过滤、中间表聚合；小表 → map join；资源不足 → 调大内存、并行度。
5. **结合调度与依赖**：是否等待上游长任务、是否被队列或资源限制拖慢。

## 8. Flink 如何实现 Exactly Once？Flink 的状态是怎么管理的？

**Exactly Once：**

- **端到端**：需要 **Source 可重放 + 内部状态一致性 + Sink 幂等或事务**。
- **内部**：基于 **Checkpoint**：JobManager 协调所有算子做 barrier 对齐的分布式快照，把 state 和 offset 持久化；失败后从上次 checkpoint 恢复，从记录的 offset 重放，保证**状态**与**处理进度**一致，即引擎侧 exactly-once 语义。
- **Sink**：Kafka 用两阶段提交（FlinkKafkaProducer 的 transactional sink）；JDBC 可用幂等写或业务主键去重；文件可用 checkpoint 提交（仅当 checkpoint 成功才可见）。

**状态管理：**

- **类型**：Keyed State（ValueState、ListState、MapState 等）、Operator State（如 list state）；还有 Broadcast State。
- **存储**：运行时在 TM 的 JVM 堆或堆外；**Checkpoint 时** 将 state 持久化到 State Backend（如 RocksDB、FS），RocksDB 支持增量 checkpoint，减轻 IO。
- **TTL**：可为 state 设置 TTL，过期自动清理，减少状态膨胀。

## 9. Kafka 异常数据你是怎么处理的？如果某个 topic 滞后严重你会怎么排查？Partition 多了或少了会对下游消费有什么影响？

**异常数据：**

- **接入前**：在 Flink/Kafka Connect 里做格式校验、schema 校验，非法数据写入死信 topic 或日志，并告警。
- **接入后**：数仓侧在 ODS/DWD 做过滤、标准化；异常值打标或落隔离表，不参与核心指标。

**Topic 滞后（lag 大）排查：**

1. 看 consumer group 的 lag 分布：是否集中在某几个 partition（倾斜）。
2. 看消费速率与生产速率：是否消费能力不足（并发、单条处理耗时、下游阻塞）。
3. 下游是否慢：写库、写 HDFS、外部 API 是否成为瓶颈。
4. 优化：提高 consumer 并发（不超过 partition 数）、优化单条处理逻辑、扩容 partition（需评估 rebalance 与顺序性）、扩容 consumer 实例。

**Partition 数量影响：**

- **过多**：小 topic 导致单 partition 数据很少，增加 broker 元数据与连接开销；大 topic 可提高并行度，但过多可能带来不必要的 rebalance 和调度成本。
- **过少**：消费并行度上不去，易成为瓶颈，lag 难降；增加 partition 可提高并发，但 rebalance 时会影响消费顺序与进度。
- **变更**：增加 partition 后，新 partition 参与分配，旧数据仍在旧 partition；减少 partition 一般不支持，需重建 topic 或双写迁移。

## 10. ClickHouse 用在什么场景？说说你对 MergeTree 表引擎的理解

**适用场景：**

-  OLAP 分析、大宽表聚合、日志/行为分析、实时报表；**高吞吐写入 + 列存压缩 + 向量化查询**，适合按维度/时间做聚合和筛选，对点查、高并发小查询不如 KV 或缓存。

**MergeTree 理解：**

- **基础**：MergeTree 是 ClickHouse 最常用的表引擎，数据按 **主键（order by）** 排序存储，按 **分区键（partition by）** 切分目录。
- **写入**：数据先写 **data part**（不可变），后台 **merge** 把多个 part 合并成更大的 part，减少 part 数量、提升读性能。
- **查询**：利用 order by 做索引裁剪，只读相关 part 和 mark；列存只读需要的列。
- **变种**：ReplacingMergeTree 按主键去重（去重发生在 merge 时）；SummingMergeTree 同主键数值自动相加；AggregatingMergeTree 存预聚合状态等，用于不同建模需求。

## 11. AIPL 模型和 RFM 模型分别适用于什么场景？一般怎么落地？

**AIPL（Awareness–Interest–Purchase–Loyalty）：**

- **适用**：偏**品牌/营销**，把人群分为认知、兴趣、购买、忠诚等阶段，用于做人群分层和营销触达（如对“认知但未转化”做投放）。
- **落地**：用行为与交易数据给用户打阶段（如曝光/点击→Interest，首单→Purchase，复购/会员→Loyalty），在数仓建用户 AIPL 标签表，供 CDP/营销平台圈选。

**RFM（Recency、Frequency、Monetary）：**

- **适用**：偏**客户价值与复购**，适合电商、会员运营，看最近一次消费、消费频次、消费金额，用于识别高价值、流失、沉睡用户。
- **落地**：从订单表算 R/F/M，按分位数或业务规则打分、分箱，生成 RFM 分或人群标签，落 DWS/ADS 或标签表，供运营策略使用。

## 12. 某短视频平台要建“高价值留存用户”标签体系，你会从哪些数据出发，怎么设计数仓和模型？

**参考答案：**

- **数据来源**：行为日志（播放、点赞、评论、分享、停留时长、完播率）、账号与设备、注册与活跃日期、付费/打赏（若有）。
- **标签定义**：“高价值”可结合 RFM 或 LTV、付费、互动深度；“留存”看 N 日留存、连续活跃周数等；可拆成“高价值 + 高留存”“高价值 + 流失风险”等子标签。
- **数仓与模型**：
  - DWD：用户行为明细（按用户、设备、天）、用户维度表。
  - DWM：用户日/周行为汇总（播放量、互动次数、时长、是否付费等）。
  - DWT：用户主题宽表（拉链或快照），含历史留存、累计价值指标。
  - DWS/ADS：按“留存周期 + 价值分档”的汇总表或标签表，如“7 日留存 + 高价值”人数、占比；可再输出到标签系统供圈选。
- **落地**：用 Hive/Flink 产出标签表或人群表，通过接口或导出到 CDP/运营平台。

## 13. CDP 中同一用户在多个渠道行为不同，如何实现统一 ID 或统一标签体系？怎么处理冲突标签？

**统一 ID：**

- **多端标识**：设备 ID、Cookie、手机号、账号 ID 等通过 **ID 映射表** 或 **图/模型** 做关联（登录关系、设备绑定、手机号归一），生成 **统一 ID（如 union_id）**，行为与标签都挂到统一 ID。
- **落地**：在 DWD/DWM 做 ID 打通（如设备+登录日志 → 映射表），下游基于统一 ID 做聚合和标签。

**统一标签体系：**

- 定义**统一标签目录**（业务口径、计算逻辑、更新周期），各渠道数据清洗后按统一口径打标，写入同一套标签表或宽表，按统一 ID 关联。

**冲突标签：**

- **规则优先**：定义优先级（如“付费用户”覆盖“未付费”、“最近一次渠道”覆盖历史）；或按时间取最新、按渠道权重加权。
- **多值并存**：若业务需要，可保留“渠道维度的标签”+“汇总标签”，由应用层按场景选择；冲突时记录来源与时间，便于审计和策略调整。

## 14. 业务要增加“7 天未激活用户”标签，落在 DWS 层，你会从哪层开始设计？指标怎么算？

**参考答案：**

- **从哪层设计**：从 **DWD 行为/激活明细** 开始，明确“激活”口径（如首登、首单、首完播）；在 **DWM** 做用户维度“最后活跃日/最后激活日”的汇总；在 **DWS** 按“统计日 - 最后激活日 ≥ 7”打“7 天未激活”标签，并做人数、占比等指标。
- **指标计算**：
  - 分母：统计日当天的“全量用户”或“曾有激活行为的用户”（按业务口径）。
  - 分子：满足“最后激活时间 &lt; 统计日 - 7 天”的用户数。
  - 可产出：7 天未激活用户数、占比、按渠道/产品等维度分布；表结构可为（dt, user_id, is_7d_inactive, …）或（dt, dimension, 7d_inactive_cnt, total_cnt, ratio）。

## 15. 用 Sqoop 导数据时，有没有遇到导入失败或字段类型不匹配？你是怎么做的？

**参考答案：**

- **导入失败**：看 Sqoop 与 MR 日志，常见有连接超时、权限、表不存在、分区不存在；处理方式包括调大超时、检查账号权限、先在目标库建好表/分区；若报 OOM，可调大 map 内存或减少并行度。
- **字段类型不匹配**：Sqoop 映射的 Hive 类型与源库不一致会导致报错或数据错误；处理方式：用 `--map-column-hive` 指定列类型，或在 Hive 端先建表再 `--hive-import` 时保证类型一致；对日期、数字、大字段要特别核对，必要时在 Hive 侧用 cast 清洗。

## 16. 对 YARN 资源调度的理解

**参考答案：**

- **角色**：ResourceManager（RM）负责集群资源分配与调度；NodeManager（NM）负责单机资源与 Container；ApplicationMaster（AM）向 RM 申请资源，在 Container 里跑 task。
- **调度器**：
  - **FIFO**：按提交顺序排队。
  - **Capacity**：队列划分资源比例，队列内可再 FIFO 或 Fair。
  - **Fair**：按需求与权重做公平共享，多任务时动态平衡资源。
- **流程**：Client 提交作业 → RM 分配 Container 启 AM → AM 向 RM 申请 Container 跑 Map/Reduce 等 task → NM 启动 Container 执行 task → 完成后释放资源。
- **资源模型**：通常用内存（vcore 可选）描述 Container 需求；队列可设 min/max 资源、ACL，用于多租户与优先级控制。

## 17. DataX 和 Sqoop 你更推荐哪个？在什么场景下选 DataX 而不是 Sqoop？

**参考答案：**

- **Sqoop**：基于 MapReduce，适合 **Hadoop 生态内** 大批量离线同步（如 RDBMS ↔ Hive/HDFS），对 Hive 集成好；强项是库到 Hive 的批同步。
- **DataX**：插件化、单机多线程，**不依赖 MR**，部署简单；插件多（RDB、Hive、HDFS、Kafka、ES 等），适合**异构数据源、中小数据量、需要灵活插件**的场景。
- **选 DataX 的场景**：不同步到 Hive 而到 Kafka/ES/其他 DB；不想起 YARN/MR；数据量不是特别大、希望低依赖、易维护；需要丰富的源端/目的端类型。选 **Sqoop** 的场景：大批量 RDBMS ↔ Hive/HDFS，且已在用 Hadoop，可接受 MR 依赖。

## 18. 一个 Hive 脚本出现数据倾斜，你怎么定位是哪个 key 倾斜？

**参考答案：**

1. **看执行计划与日志**：`explain` 看 reduce 数量；运行后看 **Job/Stage 的 counter**，如 “Bytes read/written per reduce” 或各 reduce 的 record 数，找出明显偏大的 task。
2. **加分布采样**：对 join key 或 group by key 做 `select key, count(1) as cnt from ... group by key order by cnt desc limit 100`，看哪些 key 的 count 远大于其他。
3. **用 rand 打散再聚合**：对可疑 key 先 `concat(key, '_', cast(rand()*N as int))` 做一次聚合，再对结果去前缀做二次聚合，观察中间阶段是否还有单 task 特别慢。
4. **工具**：若有监控，看各 reduce task 的输入/输出数据量分布，一眼能看出倾斜的 key 或 partition。

## 19. 有没有在 Hive 中实现过多级指标的联动，比如一个核心指标拆成多个维度指标？

**参考答案：**

- **有**。例如核心指标“GMV”拆成：**GMV = 订单数 × 客单价**，再按渠道、品类、地区等维度拆成“各渠道 GMV”“各品类 GMV”等；或 **DAU = 新登 + 老登回流**，再按来源、设备等拆。
- **实现方式**：
  - 在 DWS 建**中间表**，先算核心指标和一级维度（如总 GMV、各渠道 GMV），再基于中间表或同一 DWD/DWM 算二级维度（如渠道×品类），保证口径一致、可对账。
  - 或用 **with 子句/视图** 把“核心指标计算”写一层，多张维度表共用这一层，避免重复逻辑。
  - 指标定义和拆解关系写在文档或指标平台，便于血缘和口径统一。

## 20. 业务经常临时改需求，比如改口径要全量回溯一年数据，数仓怎么设计才能快速响应？

**参考答案：**

- **分层与主题**：DWD 保持“尽量原始、可复算”，只做清洗和标准化，不做业务口径；口径和衍生指标放在 DWM/DWS/ADS，这样改口径只需从受影响层重算，不必从 ODS 全量重推。
- **幂等与可重跑**：任务按分区/批次设计为**幂等**（同一分区多次跑结果一致），便于按日/周/月回溯；依赖调度支持“从某分区开始重跑”。
- **参数化与配置**：口径（如时间窗口、维度、过滤条件）尽量参数化或配置化，改配置即可重算，减少改代码。
- **资源与并行**：回溯时能并行跑多分区（多任务或分区并行），利用空闲资源缩短回溯时间；历史冷数据可放廉价存储，需要时再算。
- **增量与全量**：能增量的尽量增量，必须全量时只重算受影响的分区或表，避免全库全表重刷。

## 21. 有没有实际遇到数据对不上业务预期？最后怎么查出是哪个环节出问题？怎么和业务沟通的？

**参考答案（示例）：**

- **现象**：某日报表的某指标与业务系统或业务预期不一致。
- **排查**：从报表结果反推 DWS → DWM/DWD → ODS；对异常日期的分区做**逐层对账**（如 DWS 汇总与 DWM 明细 sum 对比）；检查**任务是否失败、重跑、延迟**导致用了错误分区或重复数据；对关键环节做**抽样**（如某订单是否被正确归类、是否被过滤）。
- **定位**：曾发现是**上游任务延迟**导致用了上一日分区；或**去重逻辑**漏掉某种重复；或**口径理解不一致**（如统计口径与业务侧定义不同）。
- **沟通**：先和业务对齐**指标定义与取数范围**；提供**排查过程与证据**（哪一层开始不一致、样本数据）；给出**修复方案与影响范围**（是否要重跑、回溯）；对口径歧义形成文档，避免再次偏差。

## 22. Hive 中处理 NULL 有哪些方式？Join 时 NULL 导致数据丢失怎么避免？

**处理 NULL 的方式：**

- **过滤**：`where col is not null`。
- **默认值**：`nvl(col, 0)`、`coalesce(col, '')`、`if(col is null, default, col)`。
- **条件逻辑**：`case when col is null then ... else ... end`。
- **聚合**：`count(col)` 不计 NULL，`sum(col)` 遇 NULL 视作 0（或先 nvl）；需区分“无记录”与“值为 0”时慎用 nvl。

**Join 时 NULL 导致数据丢失：**

- **原因**：标准 SQL 中 `join on a.key = b.key` 时，若 a.key 或 b.key 为 NULL，条件为 UNKNOWN，该行不会匹配，相当于被“丢”掉。
- **避免方式**：
  - **先补默认值再 join**：`on nvl(a.key, '') = nvl(b.key, '')`（若业务允许把 NULL 当同一种 key）；或 `on (a.key = b.key or (a.key is null and b.key is null))`。
  - **先过滤再 join**：若业务上 NULL 无意义，可 `where a.key is not null and b.key is not null` 再 join，但要评估是否故意丢数。
  - **用 COALESCE 生成占位 key**：如 `on coalesce(a.key, concat('null_', a.id)) = coalesce(b.key, concat('null_', b.id))`，让 NULL 也能匹配到一条（需结合业务是否合理）。
- **最佳实践**：在 DWD 层对 join key 做规范（NULL 统一处理成默认值或单独标识），下游 join 逻辑简单且不丢数。
