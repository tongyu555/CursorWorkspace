# 网络数据处理数仓分层体系总结

> 基于中台 H3 业务迁移知识库中的**表命名规范、血缘链路（`血缘链路/*.txt`）及生产 SQL**，梳理本数仓的分层定义、各层职责，并以 **CDR/VoLTE 掉话**、**5G MR 覆盖** 两条真实链路举例说明。

---

## 一、分层总览

本数仓**未完全采用互联网通用的 ODS/DWD/DWS/ADS 英文字母分层命名**，而是通过 **表名前缀 + 时间粒度后缀** 表达层级与用途。与通用数仓的对应关系如下：

| 本数仓层级 | 表名前缀/类型 | 通用数仓类比 | 核心职责 |
| :--- | :--- | :--- | :--- |
| **贴源 / 接入层** | `ods_`、`itg_` | ODS | 对接网管、DPI、CDR 等源系统，保留原始或轻度解析数据 |
| **临时 /  staging** | `temp_` | STG / 临时层 | 多源合并、IMSI 回填、中间过渡，一般不对外提供 |
| **明细 / 事实层** | `fact_` | DWD | 清洗、标准化、统一粒度的事件/会话/测量明细 |
| **中间宽表层** | `dw_`（少量） | DWM | 主题内中间加工，如工参宽表、栅格化中间结果 |
| **汇总 / 主题层** | `aggr_` | DWS | 按用户/小区/区域/栅格等维度汇聚指标 |
| **应用 / 专题层** | `dm_` | ADS / DM | 面向报表、大屏、WebGIS、规划专题、质量专题 |
| **质量稽核层** | `dq_` | 质量层 | 独立运行的数据质量校验，不阻塞主链路但并行监控 |
| **配置 / 维度** | `cfg_`、`fact_*_cm_*`、工参链 | DIM | 地市、小区工参、字典、维表属性 |

**时间粒度后缀（与 `_q/_h/_d` 等组合）**：

| 后缀 | 含义 | 示例 |
| :--- | :--- | :--- |
| `_q` | 15 分钟 / quarter | `aggr_nr_dpi_n1n2_base_q` |
| `_h` | 小时 | `fact_lte_volte_erab_release`（小时分区） |
| `_d` | 日 | `aggr_lte_usrarea_volte_calldropcause_d` |
| `_wd` / `_w` | 周 | `aggr_nr_resident_usrcel_wd` |
| `_md` / `_m` | 月 | `dm_user_perception_area_score_md` |

**下游导出任务**（非新分层，而是 dm/aggr 的**分发形态**）：`*_spark2doris`、`*_spark2pg`、`*_2hbase`、`*_2webgis`、`*_2gbase` 等。

## 二、各层功能详解

### 2.1 贴源层（`ods_` / `itg_`）

**作用**：

- 从 **CDR、DPI、MR、PM、工参（CM）** 等源系统接入数据
- 按省、日、小时等分区落地，**尽量保留源字段**，只做必要解析
- 作为全链路「单一事实来源」，下游不直连源库

**数据特点**：

- 多厂商、多接口并存（如 `ods_lte_cdr_hw_q`、`ods_lte_cdr_zx_q` 汇聚到 `ods_lte_cdr_q`）
- 可能存在重复、空小区、编码不统一等问题，**不在本层做复杂业务指标**

**典型表**：

- `ods_lte_cdr_q` — LTE CDR 统一贴源
- `ods_nr_mro_northctcc_q` — 5G MR 北向数据
- `ods_5gdpi_n1n2_e2e_telecom_5min` — 5G N1N2 控制面 DPI
- `itg_cm_nr_dx_cdu_d` — 5G 工参集成贴源

---

### 2.2 临时层（`temp_`）

**作用**：

- 多表 UNION、IMSI/用户标识回填、MR 关联等**中间步骤**
- 生命周期短，一般仅服务下游 1～2 个 fact 任务

**示例**：

- `temp_lte_cdr_imsi_mr_q`：CDR 与 MR 关联回填 IMSI，供 `fact_lte_erab_release` 等使用

---

### 2.3 明细 / 事实层（`fact_`）

**作用**：

- **DWD 核心**：清洗、过滤、字段标准化、业务规则加工
- 保持**事件/话单/测量级明细粒度**（一条 ERAB 释放、一条 MR、一条 N1N2 流程）
- 统一维度键：`cellkey`、`imsi`、`msisdn`、`provincecode` 等
- 为上层 aggr 提供可复用的明细基础

**典型加工**（以你打开的 `fact_lte_volte_erab_release` 为例）：

- 从 `fact_lte_erab_release` 筛选 **VoLTE 相关 QCI（1, 2）**
- 对 `cause` 做 **result 归类**（CASE WHEN 映射掉话原因）
- 保留无线测量、位置、时长等明细字段，写入 VoLTE 专用事实表

**数据特点**：

- 数据量最大，分区通常为 **省 + 日 + 小时**
- 一张 fact 常对应一种 **XDR/话单/测量类型**

---

### 2.4 中间层（`dw_`，少量）

**作用**：

- 主题内**中间宽表或衍生明细**，介于 fact 与 aggr/dm 之间
- 例如工参重构、栅格化中间结果

**示例**：

- `dw_lte_netmaxl_nbi_gridxy_h` — 栅格化中间表
- `dw_nr_cm_projdata_d` — 工参日快照中间加工

---

### 2.5 汇总层（`aggr_`）

**作用**：

- **DWS 核心**：在 fact 或下层 aggr 上按维度 **GROUP BY 汇聚指标**
- 维度逐级升高：`celusr`（小区+用户）→ `usrarea`（用户区域）→ `cel`（小区）→ `grd`（栅格）
- 支持 **降维汇总**（`index_reduce`）：多行合并、指标 SUM、比例重算
- 是指标复用的中枢，避免 dm 层重复扫 fact 大表

**命名规律**：

| 片段 | 含义 |
| :--- | :--- |
| `usr` / `usrcel` / `usrarea` | 用户 / 用户+小区 / 用户+区域 |
| `cel` / `celgrd` / `grd` | 小区 / 小区+栅格 / 栅格 |
| `coverage` | 覆盖类 MR 指标 |
| `dpi_n1n2` / `volte` | 业务主题 |
| `index_reduce` | 指标降维汇聚 |
| `refactor` | 模型标准化重构表 |

**数据特点**：

- 粒度比 fact 粗，比 dm 更偏「通用主题汇总」
- 常存在 **q → h → d** 的粒度下钻或汇聚关系

---

### 2.6 应用 / 专题层（`dm_`）

**作用**：

- **面向消费端**：用户感知评分、问题小区清单、规划专题、质量报表、大屏统计
- 往往 **窄表、强业务语义**，直接对接 BI、WebGIS、Doris、PostgreSQL、HBase
- 可跨多个 aggr 主题拼装（如用户感知 = 覆盖 + VoLTE + DPI 多源）

**典型场景**：

- `dm_user_perception_area_score_d` — 区域用户感知评分
- `dm_5gcm_problem_cell_info_d` — 5G 问题小区专题
- `dm_volte_usrcel_d` — VoLTE 用户+小区应用表
- `dm_45g_usrarea_kqi_e2e_d` — 45G 端到端 KQI

---

### 2.7 质量稽核层（`dq_`）

**作用**：

- **并行**于主链路，对 ods/fact/aggr 做完整性、一致性、异常清单校验
- 异常数据推送告警或写入问题清单表，**不替代主链路清洗**

**示例**：

- `dq_ods_lte_cdr_q` — CDR 贴源质量
- `dq_datasouce_volte_no_cell_list_d` — VoLTE 无小区清单
- `dq_fact_nr_mro_index_reduce_q` — MR 降维结果校验

---

### 2.8 配置 / 维度（`cfg_`、工参 fact）

**作用**：

- 提供 **join 维表**：省市县、小区工参、类目字典
- 工参链路：`itg_cm_*` → `fact_nr_cm_projdata` → 下游 aggr/dm 关联 `cellkey`

**示例**：

- `cfg_city` / `cfg_district_moca` — 地市、区县
- `fact_nr_cm_projdata` — 5G 小区工参事实（兼维度属性）

## 三、标准数据流向

```text
源系统（网管/DPI/CDR/探针）
        │
        ▼
┌───────────────────┐
│  ods_ / itg_      │  贴源接入，按分区落地
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  temp_（可选）     │  多源合并、标识回填
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  fact_            │  明细清洗、规则标准化
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  aggr_            │  分维度指标汇总（可多层）
│  （index_reduce）  │  降维、重构、主题宽表
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  dm_              │  专题应用、报表、大屏
└─────────┬─────────┘
          │
          ▼
   Doris / PG / HBase / WebGIS / GBase

          ║  并行
          ▼
┌───────────────────┐
│  dq_              │  质量稽核与异常清单
└───────────────────┘
```

**设计原则（与通用数仓一致）**：

1. **单向依赖**：下游不反向依赖 dm 算 fact
2. **明细只算一次**：指标在 aggr 汇聚，dm 做拼装而非重复扫 ods
3. **口径下沉**：核心 KPI 在 aggr 定义，dm 引用
4. **分区规范**：输入表带 `${p_provincecode}`、`${p_date}`、`${p_hour}` 等静态分区写入

## 四、实际链路示例一：CDR → VoLTE ERAB 掉话 → 用户感知

> 血缘来源：`血缘链路/CDR.txt`  
> 涉及你当前任务：`fact_lte_volte_erab_release.sql`

### 4.1 链路图

```text
ods_lte_cdr_dt_q / hw_q / zx_q / …
        │
        ▼
ods_lte_cdr_q                    ← 多厂商 CDR 统一贴源
        │
        ├── dq_ods_lte_cdr_q       ← 贴源质量稽核（并行）
        │
        ▼
temp_lte_cdr_imsi_mr_q           ← CDR 与 MR 关联，回填 IMSI
        │
        ▼
fact_lte_erab_release            ← ERAB 释放明细（全 QCI）
        │
        ▼
fact_lte_volte_erab_release      ← 筛选 QCI in (1,2)，VoLTE 掉话原因归类
        │
        ▼
aggr_4g_volte_calldrop_causecelusr_q   ← 按小区+用户+掉话原因汇总（15min）
        │
        ▼
aggr_lte_usrarea_volte_calldropcause_cel_d  ← 用户区域+小区+原因 日汇总
        │
        ▼
aggr_lte_usrarea_volte_calldropcause_d       ← 用户区域级日汇总
        │
        ├── aggr_lte_usrarea_volte_calldropcause_d_2hbase  → HBase
        │
        └── （与同主题 RRC/ERAB 等 aggr 汇合）
                    │
                    ▼
            dm_user_perception_area_score_d/md/wd   ← 区域用户感知评分（应用层）
                    │
                    └── dm_user_perception_area_score_md_2webgis → WebGIS 地图
```

### 4.2 各层在本链路中的具体作用

| 层级 | 表 | 本链路中做什么 |
| :--- | :--- | :--- |
| ODS | `ods_lte_cdr_q` | 接入多厂商 CDR 原始话单 |
| TEMP | `temp_lte_cdr_imsi_mr_q` | 关联 MR，补全用户标识 |
| FACT | `fact_lte_erab_release` | ERAB 释放事件明细 |
| FACT | `fact_lte_volte_erab_release` | **VoLTE 子集 + cause→result 业务规则** |
| AGGR | `aggr_4g_volte_calldrop_causecelusr_q` | 掉话次数/原因按小区用户汇聚 |
| AGGR | `aggr_lte_usrarea_volte_calldropcause_*` | 逐级升维到用户区域 |
| DM | `dm_user_perception_area_score_*` | 合成感知评分，供运营分析 |
| DQ | `dq_ods_lte_cdr_q` | 监控贴源是否缺数、异常 |

### 4.3 关键代码片段（fact 层业务规则）

`fact_lte_volte_erab_release` 从通用 ERAB 释放事实中**派生 VoLTE 主题事实**：

```sql
-- 来源：fact_lte_erab_release
-- 过滤：qci in (1, 2)  -- VoLTE 承载
-- 加工：cause 映射为 result（掉话归类）
FROM dtsw_db.fact_lte_erab_release
WHERE qci in (1, 2)
```

体现 **fact 层「清洗 + 主题拆分」**：同一 ODS 明细，按业务（VoLTE / 数据业务）拆成不同 fact，供不同 aggr 复用。

## 五、实际链路示例二：5G MR 覆盖 → 用户感知 → 大屏

> 血缘来源：`血缘链路/5G_MR.txt`

### 5.1 链路图（精简主链）

```text
ods_nr_mro_northctcc_temp_new
        │
        ▼
ods_nr_mro_northctcc_location → ods_nr_mro_northctcc_q    ← MR 贴源+定位
        │
        ├── dq_ods_nr_mro_northctcc_location
        │
        ▼
aggr_nr_coverage_usrcelgrd_q          ← 用户+小区+栅格 覆盖汇总（15min）
        │
        ▼
aggr_nr_coverage_base_q / base_h      ← 覆盖基础指标
        │
        ├── aggr_nr_coverage_index_reduce_usrcelgrd_q/h   ← 降维汇聚
        │         │
        │         ▼
        │   aggr_nr_coverage_resident_usrgrd_d
        │         │
        │         ▼
        │   aggr_nr_usr_perception_usrcelarea_d          ← 用户感知（覆盖主题）
        │         │
        │         └── 与 DNS/IM/Video/VoNR 等 aggr 并列汇合
        │
        ├── aggr_nr_coverage_celgrd_h → celgrd_d → cel_d   ← 小区/栅格日表链
        │         │
        │         ├── aggr_nr_coverage_index_reduce_cel_d
        │         ├── dm_5gmr_cell_list_d
        │         └── dm_5gcm_cover_cellgrid_d → spark2doris
        │
        └── fact_nr_mro_neighbor_nr_q → problem 类 aggr   ← 邻区/问题小区专题
                    │
                    ▼
              dm_5gcm_problem_cell_info_d
                    │
                    ├── dm_5gcm_big_screen_stat_d → spark2doris   ← 大屏
                    └── spark2webgis
```

### 5.2 本链路的分层特点

1. **ODS 层**：MR 北向数据体积大，先 location 再 q，保证定位后再入标准分区表  
2. **AGGR 层为主力**：MR 场景 rarely 长期保留全量 fact，多在 `aggr_nr_coverage_*` 系列按 **usrcelgrd / cel / grd** 多级汇聚  
3. **index_reduce 子层**：在 aggr 内部再做**降维**（多 plmn、多数据源合并），对应模型标准化文档中的 PDM 列名重构  
4. **DM 层**：`dm_5gcm_*` 面向 **网优/规划/大屏**，`dm_5gmr_*` 面向 **MR 清单类应用**  
5. **多主题汇合**：`aggr_nr_usr_perception_usrcelarea_d` 与 DPI（web/im/video/vonr）aggr 共同支撑端到端用户感知

## 六、与通用 ODS/DWD/DWS/ADS 对照记忆

| 面试/对外表述 | 本数仓说法 | 一句话 |
| :--- | :--- | :--- |
| ODS | `ods_` / `itg_` | 源系统什么样，贴源就什么样 |
| DWD | `fact_`（+ 少量 `temp_`） | 明细干净、粒度统一、能复用 |
| DWS | `aggr_` | 指标算好、按主题和维度汇好 |
| ADS/DM | `dm_` | 给报表、大屏、专题直接用 |
| 质量 | `dq_` | 跑在旁边的「安检」 |
| 维度 | `cfg_` / 工参 `fact_*_cm_*` | 给事实表补全属性 |

---

## 七、面试 / 汇报口述模板（1 分钟）

> 我们数仓按 **ods 贴源 → temp 过渡 → fact 明细 → aggr 汇总 → dm 应用** 分层，并行 **dq 质量** 和 **cfg/工参维度**。  
> 以 VoLTE 掉话为例：CDR 进 `ods_lte_cdr_q`，关联 MR 后形成 `fact_lte_erab_release`，再筛 QCI 产出 `fact_lte_volte_erab_release`，在 aggr 层按小区、用户区域汇聚掉话原因，最终汇入 `dm_user_perception_area_score` 做区域感知评分，并同步 WebGIS。  
> 5G MR 链路类似，但 aggr 层更重，覆盖 usrcelgrd、cel、grd 多级粒度，经 index_reduce 降维后供感知专题和大屏使用。
