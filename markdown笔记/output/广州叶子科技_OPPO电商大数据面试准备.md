## 一、SQL 高频题（必考：手写 + 优化）

### 1.1 窗口函数（重中之重）

#### Q：窗口函数有哪些？区别与场景？

| 类型 | 函数 | 特点 | 典型场景 |
| :--- | :--- | :--- | :--- |
| 排序类 | `row_number()` | 连续序号，同值不同号 | 取 TopN、去重取最新一条 |
| | `rank()` | 同值同号，后续跳号 | 排行榜（并列有空缺） |
| | `dense_rank()` | 同值同号，后续不跳号 | 密集排名、评奖并列名次 |

**精简背诵**：

- `row_number()`：连续序号，同值不同号，无跳号 → **用户单日消费 Top10**
- `rank()`：同值同号，后续序号跳号 → **排名展示、评奖并列**
- `dense_rank()`：同值同号，后续序号不跳号 → **分段排名无空缺**

**OPPO/电商举例**：

- 每个用户**最近 3 笔订单** → `row_number() over(partition by user_id order by order_time desc)`
- 商品销量 **Top10** → `row_number() over(order by sale_cnt desc) <= 10`
- 用户**连续登录** → 见下题

#### Q：如何查询连续登录 N 天的用户？（经典）

**核心思路**：`日期 - row_number()` 得到分组标识，连续日期差值一致即为同一组。

```sql
-- 思路模板（Hive/Spark SQL）
SELECT user_id
FROM (
    SELECT
        user_id
        ,login_date
        ,date_sub(login_date, rn) AS grp_flag
    FROM (
        SELECT
            user_id
            ,login_date
            ,row_number() OVER (
                PARTITION BY user_id
                ORDER BY login_date
            ) AS rn
        FROM (
            SELECT DISTINCT user_id, login_date
            FROM dwd_user_login_di
            WHERE dt >= '2024-01-01'
        ) t1
    ) t2
) t3
GROUP BY
    user_id
    ,grp_flag
HAVING count(1) >= 7
;
```

> Spark 3.x 若对窗口结果过滤，外层改用 `WHERE rn = 1` 子查询写法。

#### Q：`PARTITION BY` 与 `GROUP BY` 区别？

| | GROUP BY | PARTITION BY |
| :--- | :--- | :--- |
| 行数 | 聚合后变少，一组一条 | 行数不变，每行保留 |
| 用途 | 汇总统计 | 排名、累计、偏移等行间计算 |
| 组合 | 常与聚合函数 | 常与 OVER 窗口函数 |

**精简背诵**：`group by` 分组聚合、减少行数；`partition by` 开窗分区、保留原行，用于行间计算与排名。

---

| 类型 | 函数 | 特点 | 典型场景 |
| :--- | :--- | :--- | :--- |
| 聚合类 | `sum/avg/count over()` | 累计/滑动聚合 | 累计 GMV、7 日滚动 UV |
| 偏移类 | `lag()` / `lead()` | 上/下一行 | 环比、前后行为对比 |

### 1.2 电商核心指标 SQL

#### Q：PV、UV、DAU、MAU、新增、次日留存怎么算？

**精简背诵**：

- **PV**：页面浏览总次数，`count(1)`，重复访问累加
- **UV / DAU**：独立用户数，`count(distinct user_id)`，单日一人多次只计 1 次
- **新增**：当日首次出现（活跃口径：历史无活跃；或注册日口径）
- **留存**：基准日 UV 中次日/7 日仍活跃占比，分母是**基准日 UV**

| 指标 | 口径 | SQL 思路 |
| :--- | :--- | :--- |
| **PV** | 页面浏览次数 | `count(1)` 或 `sum(pv)` |
| **UV** | 独立访客 | `count(distinct user_id)` |
| **DAU** | 日活跃用户 | 当日 `count(distinct user_id)` |
| **MAU** | 月活跃用户 | 当月去重用户 |
| **新增用户** | 首次出现 | 左关联历史用户表，历史为空即为新 |
| **次日留存** | D1 留存 | 当日活跃 ∩ 次日仍活跃 / 当日活跃 |

**DAU / 新增核心逻辑（口述）**：

- DAU：按日期 `group by`，`count(distinct user_id)`
- 新增：左联用户首次时间表，筛选当日首次出现（`history.user_id is null`）

```sql
-- 次日留存率示例
SELECT
    t1.dt
    ,count(DISTINCT t1.user_id) AS dau
    ,count(DISTINCT t2.user_id) AS retain_uv
    ,count(DISTINCT t2.user_id) / count(DISTINCT t1.user_id) AS d1_retain_rate
FROM dwd_user_active_di t1
LEFT JOIN dwd_user_active_di t2
ON
    t1.user_id = t2.user_id
    AND t2.dt = date_add(t1.dt, 1)
WHERE
    t1.dt = '2024-06-01'
GROUP BY
    t1.dt
;
```

#### Q：电商 GMV、客单价、转化率口径？

| 指标 | 口径 | 注意点 |
| :--- | :--- | :--- |
| **GMV** | 下单金额汇总（含未支付/取消需看业务定义） | 是否含退款、是否按支付口径 |
| **实付 GMV** | 实际支付成功金额 | 通常 `pay_status = success` |
| **客单价** | GMV / 下单用户数 | 分母是下单 UV 还是支付 UV 需统一 |
| **转化率** | 下一环节 UV / 上一环节 UV | 漏斗各环节分子分母要一致 |
| **复购率** | 二次及以上购买用户 / 总购买用户 | 需定义统计周期 |
| **ARPU** | 总收入 / 活跃用户数 | 常用于用户价值评估 |

#### Q：漏斗模型如何实现？（首页→详情→加购→下单→支付）

**思路**：

1. 各行为节点打宽到用户粒度（是否发生各节点）
2. 按用户统计链路完整性
3. 分层计算各环节转化率

```sql
-- 简化思路：用户日粒度行为宽表
SELECT
    count(DISTINCT CASE WHEN is_home = 1 THEN user_id END) AS home_uv
    ,count(DISTINCT CASE WHEN is_detail = 1 THEN user_id END) AS detail_uv
    ,count(DISTINCT CASE WHEN is_cart = 1 THEN user_id END) AS cart_uv
    ,count(DISTINCT CASE WHEN is_order = 1 THEN user_id END) AS order_uv
    ,count(DISTINCT CASE WHEN is_pay = 1 THEN user_id END) AS pay_uv
FROM dws_user_funnel_di
WHERE dt = '2024-06-01'
;
-- 转化率 = 下一环节 / 上一环节
```

**进阶**：严格漏斗要求**有序链路**（先首页再详情…），可用 `collect_list` + 时间排序或 Flink 实时漏斗。

**精简背诵**：按用户分组标记各行为节点 → 统计每层 UV → 逐层算转化率（详情/首页、加购/详情…）。

#### Q：统计每个用户最近 3 笔订单、Top10 商品

```sql
-- 每用户最近 3 笔订单
SELECT user_id, order_id, order_amt, order_time
FROM (
    SELECT
        user_id
        ,order_id
        ,order_amt
        ,order_time
        ,row_number() OVER (
            PARTITION BY user_id
            ORDER BY order_time DESC
        ) AS rn
    FROM dwd_order_detail_di
    WHERE dt = '2024-06-01'
) t
WHERE rn <= 3
;

-- 商品销量 Top10
SELECT product_id, sale_cnt
FROM (
    SELECT
        product_id
        ,sum(sale_cnt) AS sale_cnt
        ,row_number() OVER (ORDER BY sum(sale_cnt) DESC) AS rn
    FROM dws_product_sale_di
    WHERE dt = '2024-06-01'
    GROUP BY product_id
) t
WHERE rn <= 10
;
```

### 1.3 SQL 性能优化 & 数据倾斜（高频难点）

#### Q：Hive/Spark 大表 Join 数据倾斜，现象、原因、方案？

| 维度 | 内容 |
| :--- | :--- |
| **现象** | 任务卡住、个别 Task/Reducer 极慢、跑批超时 |
| **原因** | NULL/空值过多、热点 Key（爆款商品/大 V 用户）、分布不均 |
| **方案** | ① 过滤无效空值 ② 热点 Key 加盐打散 + 二次聚合 ③ 小表 Broadcast/MapJoin ④ 倾斜 Key 单独抽取计算再合并 |

**精简背诵**：现象=个别 Task 极慢；原因=NULL/热点 Key/两表量差大；方案=过滤空值、加盐、MapJoin、倾斜单独算。

**电商/广告热点例子**：大促爆款 SKU、头部广告位 ID、热门渠道 → Join 倾斜 → 加盐或单独分支。

#### Q：大数量级 `count(distinct)` 如何优化？

1. 内层 `group by user_id` 去重，外层 `count(1)`（减少 Shuffle）
2. `approx_count_distinct` 近似去重（允许小幅误差，如大屏 UV）
3. Bitmap 位图函数精准去重（性能较优，需环境支持）

#### Q：分区与分桶的作用与区别？

| | 分区（Partition） | 分桶（Bucket） |
| :--- | :--- | :--- |
| **层级** | 目录级别 | 文件级别 |
| **原理** | 按字段值分目录 | 按 Hash 分文件 |
| **目的** | 裁剪扫描范围 | 优化 Join、抽样 |
| **实践** | 按天分区 `dt`（必做） | 大表高频 Join 列分桶 |

#### Q：小文件问题？

**原因**：分区过多、Reduce 数过大、频繁 insert、任务拆分过细。

**危害**：NameNode 元数据压力、查询启动慢、IO 效率低。

**解决**：`concatenate` 合并、调整 Reduce/`spark.sql.shuffle.partitions`、CombineInputFormat、规范写入逻辑、AQE 自适应合并。

## 二、数据仓库专题（核心考察）

### 2.1 数仓分层

#### Q：ODS / DWD / DWS / DM（ADS）各层作用？

| 层级 | 作用 | 数据特点 | OPPO 业务举例 |
| :--- | :--- | :--- | :--- |
| **ODS** | 原始数据接入 | 贴近源系统，少清洗 | 埋点原始日志、广告日志、MySQL 订单 binlog |
| **DWD** | 明细清洗 | 去重、补全、统一粒度 | 行为明细、广告曝光/点击明细、订单/支付明细 |
| **DWS** | 轻度/主题汇总 | 按用户/设备/渠道/广告位聚合 | 用户日行为、广告转化汇总、商品日销量 |
| **DM/ADS** | 应用指标层 | 面向报表/看板/大屏 | 留存报表、广告 ROI、GMV 日报、运营专题 |

**精简背诵**：

- ODS：同步业务库与埋点，几乎不清洗，保留原貌
- DWD：清洗脏数据、去重、补字段，保留明细粒度
- DWS：按用户/渠道/设备等维度轻度聚合，中间指标复用
- DM/ADS：面向报表、大屏、专题，产出最终业务指标

#### Q：主题域如何划分？为什么这么设计？

**精简背诵**：按 **用户、设备、行为、广告、渠道** 划分主题域；拆分后职责清晰、减少耦合、方便团队协作与指标复用。

| 主题域 | 典型表/指标 | OPPO 场景 |
| :--- | :--- | :--- |
| 用户 | 用户维、活跃、留存 | DAU、新增、画像标签 |
| 设备 | 设备维、机型、OS | ColorOS 版本、设备行为 |
| 行为 | 埋点明细、路径、漏斗 | 页面浏览、点击链路 |
| 广告 | 曝光/点击/转化 | 广告 ROI、投放效果 |
| 渠道 | 来源、投放、归因 | 渠道转化、获客成本 |

#### Q：为什么不直接用 ODS 做报表？

- 原始数据脏、格式乱、重复多，查询性能差
- 分层实现：**解耦、复用、统一口径、降低维护成本**

**数据流向**：

```text
埋点/业务库/广告日志 → Kafka/DataX → ODS → DWD → DWS → DM/ADS → 报表/BI/Doris
```

### 2.2 数据建模

#### Q：维度建模、星型 vs 雪花？

**精简背诵**：

- **星型**：维度表直接关联事实表，关联少、查询快 → **工作主流**
- **雪花**：维度继续拆分多级，冗余低但链路长、性能差 → **极少用**

#### Q：事实表 vs 维度表？业务例子

| 类型 | 存储内容 | OPPO/电商例子 |
| :--- | :--- | :--- |
| **事实表** | 行为、事件、度量值 | 用户点击日志、**广告曝光/点击**、订单表、支付流水 |
| **维度表** | 描述性属性 | 用户维、**设备维**、渠道维、广告位维、时间维 |

#### Q：缓慢变化维（SCD）了解吗？

维度属性随时间变更即为缓慢变化维（SCD）：

- **SCD1**：直接覆盖旧值，只保留最新（如用户昵称）
- **SCD2**：保留历史，新增记录，用起止时间区分生命周期（如会员等级、投放策略变更）→ **用户画像/维表常用**

#### Q：如何设计一张用户行为明细表？

| 设计项 | 建议 | 说明 |
| :--- | :--- | :--- |
| **核心字段** | `user_id` / `device_id`、行为类型、行为时间、页面 ID、渠道、设备型号、OS | 支撑漏斗、路径、画像 |
| **分区** | 按天 `dt` 为主，高实时场景可加小时分区 | 缩小扫描范围，便于补数 |
| **分桶** | 高频关联字段（`user_id` / `device_id`）分桶 | 优化多表 Join、抽样查询 |

**OPPO 举例**：`dwd_user_behavior_di` 按 `dt` 分区，`user_id` 分桶，关联 `dim_device_df` 补全机型与系统版本。

### 2.3 数据治理 & 质量

#### Q：数据质量监控维度？

| 维度 | 监控内容 | 电商举例 |
| :--- | :--- | :--- |
| **完整性** | 空值、缺失分区 | 订单表 `order_id` 为空 |
| **准确性** | 逻辑校验 | 支付金额 ≤ 订单金额 |
| **唯一性** | 主键重复 | 同一订单号多条记录 |
| **及时性** | 数据延迟 | T+1 报表 8 点前未产出 |
| **一致性** | 上下游指标对齐 | DWS GMV 与 DM 报表 GMV 不一致 |

**实现**：定时执行校验 SQL + 接入监控平台 + 异常消息推送 + 问题第一时间修复

**精简背诵**：四大维度——完整性、准确性、唯一性、及时性。

#### Q：数据血缘、元数据在工作中起什么作用？

| 类型 | 作用 | 典型场景 |
| :--- | :--- | :--- |
| **元数据** | 记录表结构、字段含义、分区、任务调度信息 | 统一口径、新人上手、口径核对 |
| **数据血缘** | 追踪上下游链路 | 指标异常/任务报错快速定位源头；表结构变更前评估影响范围 |

#### Q：数据生命周期 / 冷热数据？

- **热数据**（近 30 天）：正常存储，高频查询
- **冷数据**（半年以上）：压缩、归档至低成本存储
- 按业务规则设置分区 TTL，自动清理超期数据
- **保留策略参考**：日志类 3～6 个月；核心交易/广告数据 1 年以上
- 电商：大促前扩容，大促后归档历史明细

### 2.4 业务数仓差异（叶子科技核心）

#### Q：用户行为 / 交易 / 广告 数仓设计差异？

| 维度 | 行为数仓 | 交易/电商数仓 | 广告数仓 |
| :--- | :--- | :--- | :--- |
| 数据量 | 极大（埋点） | 相对小 | 大（曝光明细） |
| 实时性 | 偏高 | T+1 为主 | 准实时/实时投放监控 |
| 核心指标 | 留存、漏斗、路径、画像 | GMV、订单、退款 | 曝光、点击、转化、ROI |

## 三、大数据组件（Hadoop / Spark / Flink / Kafka）

### 3.1 Hadoop & HDFS

| 组件 | 职责 |
| :--- | :--- |
| **HDFS** | 分布式存储，NameNode 管元数据，DataNode 存数据块，默认 3 副本 |
| **YARN** | 集群资源调度，管理 Container |
| **MapReduce** | 分布式计算（已被 Spark 大量替代，了解即可） |

**口述要点**：主从架构；NameNode 管理元数据、目录、块信息；DataNode 存真实数据块；YARN 负责资源调度。

**精简背诵 — Hadoop 三大组件**：HDFS 分布式存储；YARN 资源调度；MapReduce 离线计算（了解即可）。

---

### 3.2 Spark（离线核心，重点）

#### Q：Spark vs MapReduce 优势？

**精简背诵**：内存计算、中间结果少落盘；DAG 调度链路短；算子与 API 更丰富。

#### Q：RDD / DataFrame / Dataset？

**精简背诵**：RDD 无 Schema 弱类型；DataFrame 有 Schema 弱类型；Dataset 强类型兼顾性能；均为惰性执行。

#### Q：Spark Shuffle 原理与优化？

**精简背诵**：Shuffle = 跨分区重分发（Map 端写数据 + Reduce 端拉取数据两阶段）。优化：合理分区数（避免过多小文件或过少单 Task 压力）、开启 Shuffle 压缩、优化分区字段规避热点 Key、调整缓冲区与拉取重试参数。

#### Q：Spark 和 Hive 执行同一任务，性能差异在哪？什么场景优先 Spark？

| 维度 | Spark | Hive（MapReduce） |
| :--- | :--- | :--- |
| 计算模式 | 内存计算，中间结果少落盘 | 多轮落盘，链路长 |
| 适用场景 | 大数据量 ETL、复杂聚合、多轮计算、迭代 | 纯简单查询、小批量数据 |
| 性能 | 复杂 SQL、大表计算明显更优 | 轻量查询够用 |

**精简背诵**：大数据量 ETL、复杂 SQL 优先 Spark；简单查询、小数据可用 Hive。

#### Q：大表 Join 除数据倾斜外，还有哪些优化？

1. 小表优先 **Broadcast Join**，避免全局 Shuffle
2. 提前过滤无用数据、裁剪字段，减少传输量
3. 关联字段做 **分桶**，利用桶表 Join
4. 拆分超大任务，分批次执行

#### Q：Spark 运行架构？

**精简背诵**：Driver 解析任务、生成 DAG、调度 Task；Executor 运行 Task 并提供内存存储。

---

### 3.3 Flink（实时基础）

#### Q：Flink vs Spark Streaming？

**精简背诵**：Flink 纯流式、延迟更低；原生 EventTime/Watermark/状态管理；Exactly-Once 更稳定 → 适合**实时大屏、实时广告统计**。

#### Q：EventTime / ProcessingTime / Watermark？

**精简背诵**：ProcessingTime=系统时间；EventTime=业务发生时间（统计更精准）；Watermark=标记「该时间前数据已到齐」，触发窗口、处理乱序。

**迟到数据处理**：设置窗口允许迟到时间；超期数据写入**侧输出流**单独处理。

#### Q：窗口类型？

**精简背诵**：滚动窗口无重叠（每小时 UV）；滑动窗口有重叠（每 10 分钟统计近 1 小时）；会话窗口按空闲间隔（行为会话）。

#### Q：Flink Checkpoint 作用？生产环境如何配置？

**精简背诵**：定期保存算子状态快照，集群故障可恢复，保障 **Exactly-Once** 语义。

**生产配置要点**：按业务延迟要求设置间隔；开启**增量 Checkpoint**；配置外部持久化存储；搭配 **Savepoint** 用于版本迭代与任务重启。

---

### 3.4 Kafka（数据接入层）

#### Q：Kafka 分区、副本作用？

**精简背诵**：分区提升并行读写与吞吐；副本保证高可用，主副本故障由从副本接管。

#### Q：如何保证消息不丢失、不重复？

| 目标 | 方案 |
| :--- | :--- |
| **防丢失** | 生产端 ACK=all + 重试；存储端多副本；消费端手动提交 offset |
| **防重复** | 业务层幂等：唯一主键去重、Upsert |

#### Q：Kafka 消息积压如何排查？

1. 查看分区消费延迟（Lag）、消费组状态
2. 排查消费逻辑慢、下游写入瓶颈
3. 检查分区数与消费线程/并行度是否匹配
4. 评估是否需要扩容 Consumer、优化 Flink/Spark 消费任务

**二面展开**：① 监控面板确认分区消费延迟（Lag）② 消费端：逻辑卡顿、线程不足、消费组配置 ③ 上下游：生产流量突增、网络 IO 瓶颈 ④ 临时：加消费线程、拆分区；长期：优化消费逻辑

## 四、业务 & 工程实践（OPPO 行为 / 广告 / 电商）

### 4.1 完整数据流（必背）

```text
埋点日志 / MySQL 业务库 / 第三方数据
    → 采集（Flume/DataX/SeaTunnel）
    → Kafka 缓冲

    → DM/ADS 指标层
    → 报表 / 大屏 / Doris 即席查询
```

### 4.2 指标异常下跌排查（高频压力题）

适用：**DAU、广告转化率、GMV** 等核心指标突降。

**精简背诵 SOP**：

1. 确认指标口径是否变更
2. 排查链路：采集 → Kafka → 同步任务是否延迟/报错
3. 校验质量：脏数据、空值、重复、丢数
4. 对接业务：版本更新、活动下线、渠道/投放策略变化
5. 定位根因 → 修复 → 复盘

**标准 SOP（展开）**：

1. **确认口径**：指标定义、过滤条件、时间范围是否变更
2. **查链路**：采集 → Kafka 积压？→ ODS 分区是否缺失？→ 调度任务是否失败/延迟？
3. **查质量**：空值暴增、重复、脏数据、分区数据量异常
4. **查业务**：APP 版本升级埋点变更、活动下线、渠道切换、大促结束
5. **定位修复 + 补数 + 复盘**：记录根因，完善监控

### 4.3 离线跑批延迟排查

**精简背诵**：资源不足、数据倾斜、小文件过多、上游依赖阻塞、单日数据量突增、SQL 效率低。

| 方向 | 检查点 |
| :--- | :--- |
| 资源 | YARN 队列满、Executor 不足 |
| 性能 | 数据倾斜、小文件、SQL 未分区裁剪 |
| 依赖 | 上游任务未完成、血缘阻塞 |
| 数据量 | 大促/活动导致分区暴增 |
| 调度 | DS/Airflow 实例堆积、优先级配置 |

### 4.4 大促 / 上线期保障

**精简背诵**：集群扩容、提高任务优先级、优化 SQL/Shuffle、拆分大任务、加强监控告警、重点任务专人值守。

- [ ] 核心任务资源预留 / 队列隔离
- [ ] 热点 Key 倾斜预案（加盐、广播 Join）
- [ ] 上游 Kafka 消费延迟监控
- [ ] 关键指标质量校验 SQL 提前跑通
- [ ] 降级方案：非核心报表延后、核心 GMV/转化优先
- [ ] 值班与告警链路确认
- [ ] 预留兜底方案，出问题快速回滚

### 4.5 埋点数据丢数、重复如何治理？

| 环节 | 手段 |
| :--- | :--- |
| 客户端 | 规范上报逻辑，本地缓存与重试，减少丢数 |
| 接入层 | 基于唯一标识去重 |
| 监控 | 全链路上报量监控，波动及时预警 |
| 规范 | 统一埋点协议，定期核对埋点字典，下线无效埋点 |

### 4.6 广告业务：曝光-点击-转化链路对齐

**题目**：广告曝光、点击、转化三条链路如何保证数据对齐？

**精简背诵**：

- 统一 **设备 ID / 用户 ID** 作为关联主键
- 对齐 **时间口径**（曝光/点击/转化的归因窗口）
- 清洗异常数据（重复曝光、无效点击）
- 链路全流程监控，各环节 UV/转化率可交叉校验

### 4.7 离线 vs 实时数仓区别

| 维度 | 离线数仓 | 实时数仓 |
| :--- | :--- | :--- |
| 引擎 | Hive / Spark | Flink |
| 延迟 | T+1 / 小时级 | 秒 / 分钟级 |
| 分层 | 完整 ODS→DWD→DWS→ADS | 链路简化，侧重实时宽表 |
| 场景 | 报表、对账、标签批更新 | 实时大屏、实时告警、实时推荐特征 |
| 精度 | 高，可回溯 | 准实时，需容忍乱序 |

**精简背诵**：离线 Hive+Spark、T+1、分层完整、日报周报；实时 Flink+Kafka、秒/分钟级、分层简化、大屏与监控。

#### Q2：项目中遇到过最棘手的问题？怎么排查和解决的？

二面最爱追问细节，**优先准备以下两类真实场景**：

**场景一：数据倾斜**

| 步骤 | 内容 |
| :--- | :--- |
| 现象 | 大表 Join 个别 Task 极慢，任务卡住 |
| 定位 | 日志 + 任务监控定位倾斜 Key |
| 处理 | 空值提前过滤；热点 Key 加盐打散，拆分计算后合并；大 Join 小改 MapJoin |
| 结果 | 任务运行时长缩短 XX%（尽量量化） |

**场景二：离线任务大面积延迟**

| 步骤 | 内容 |
| :--- | :--- |
| 排查顺序 | ① 查依赖链路，上游是否阻塞 ② 集群资源、分区数据量是否暴涨 ③ SQL 低效逻辑、小文件过多 |
| 处理 | 合并小文件、优化 Shuffle、调整并行度、拆分大任务 |
| 结果 | 核心任务恢复 SLA，如 8 点前产出 |

> 更多排查 SOP 见 [4.2 指标异常](#42-指标异常下跌排查高频压力题)、[4.3 跑批延迟](#43-离线跑批延迟排查)。

#### Q3：数仓如何做分层和主题域划分？为什么这么设计？

**标准四层 + 五主题域**（口述版）：

```text
ODS  → 原始同步，不做清洗，保证可回溯
DWD  → 清洗脏数据、去重、补全字段，保留明细粒度
DWS  → 按用户/渠道/时间等轻度聚合，中间指标复用性最强
ADS  → 面向报表、大屏、业务分析，产出最终指标

主题域：用户 / 设备 / 行为 / 广告 / 渠道
```

**设计理由**：职责清晰、减少耦合、方便团队协作与指标复用。

> 详见 [2.1 数仓分层](#21-数仓分层)、[主题域划分 Q](#q主题域如何划分为什么这么设计)。

**二面口述要点**：核心字段（用户/设备 ID、行为类型、时间、页面、渠道、机型、OS）+ 按天分区（必要时加小时）+ 高频关联字段分桶。

### 6.6 业务场景题（OPPO 专属，高频）

#### Q：DAU / 广告转化率突然下跌，完整排查流程？

**按顺序答**（二面常要求完整 SOP）：

1. **核对指标口径**：统计逻辑、字段、时间范围是否近期修改
2. **排查数据链路**：埋点采集 → Kafka → 同步任务是否延迟、报错、丢数
3. **校验数据质量**：脏数据、空值、重复数据是否暴增
4. **对接业务侧**：APP 版本更新、活动下线、渠道投放调整、新品动作
5. **定位根因 → 修复 → 复盘**

> 详见 [4.2 指标异常排查](#42-指标异常下跌排查高频压力题)。

#### Q：大促 / 版本上线期间，如何保障数据链路稳定？

- 集群资源提前扩容，调高核心任务资源配额与优先级
- 优化 SQL、Shuffle 逻辑，提前压测
- 全链路监控与告警，专人值守
- 拆分超大任务，避免单点故障
- 预留兜底方案，出问题快速回滚
