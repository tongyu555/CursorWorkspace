# 需要熟悉 · 详细知识点（前三模块）

> 对应《大数据开发-系统学习大纲》中「二、需要熟悉」的前三个标题，供系统学习与面试准备使用。

---

## 一、多模态数据与特征工程

### 1.1 多模态数据类型（详细）

#### 1.1.1 视觉数据

| 知识点 | 说明 |
|--------|------|
| **数据来源** | 车内摄像头：DMS（驾驶员）、OMS（乘员）、环视/舱内全景；部分车型含手势/视线摄像头 |
| **数据形态** | 图像（单帧）、视频流（按帧或按段存储）；常见格式 JPEG/PNG、H.264/H.265 编码视频 |
| **存储与采样** | 按时间/事件采样（如每 5 秒一帧、检测到闭眼时触发）；存储路径、时间戳、设备 ID、关联会话 ID |
| **与数仓的关系** | 元数据入数仓（路径、时间、标签）；原始媒体多在对象存储，数仓存索引与标注结果、特征表 |
| **清洗要点** | 无效帧过滤（黑屏、过曝）、时间戳对齐、与车辆/行程维度关联；隐私脱敏（人脸区域打码或仅存特征） |

#### 1.1.2 语音数据

| 知识点 | 说明 |
|--------|------|
| **数据来源** | 麦克风阵列、语音助手交互、电话/会议音频 |
| **数据形态** | 波形（PCM/WAV）或压缩格式（AAC/OPUS）；通常带采样率、声道、时长 |
| **存储与采样** | 按会话/唤醒事件切分；关联文本（ASR 结果）、说话人、情绪等标签 |
| **与数仓的关系** | 音频路径与元数据、转写文本、声学特征（可预计算入库）；大文件走对象存储，数仓做索引与统计 |
| **清洗要点** | 静音段裁剪、采样率/格式统一、背景噪声标注；与业务事件（如导航、音乐）的时间对齐 |

#### 1.1.3 文本数据

| 知识点 | 说明 |
|--------|------|
| **数据来源** | 语音转写（ASR）、用户输入、导航/音乐/车控指令、日志与告警文本 |
| **数据形态** | 结构化字段（指令类型、槽位） + 非结构化原文；可能含多轮对话序列 |
| **存储与采样** | 按会话/请求存储；需会话 ID、轮次、时间戳、角色（用户/系统） |
| **与数仓的关系** | 直接入表；做分词、意图分类、实体抽取后的结果表；用于统计、标注与模型训练 |
| **清洗要点** | 敏感词过滤、长度与编码统一、会话边界与去重；多轮对话需保持轮次与上下文顺序 |

#### 1.1.4 车辆时序数据

| 知识点 | 说明 |
|--------|------|
| **数据来源** | CAN/LIN 信号、传感器（车速、转向、加速度）、GPS、电池/电机等电控数据 |
| **数据形态** | 时间序列：时间戳 + 信号 ID/名称 + 物理值（数值/枚举）；采样频率从 10Hz 到 1kHz 不等 |
| **存储与采样** | 按行程/时间窗口落库；常见列式或时序库；需车辆 ID、行程 ID、信号定义（DBC/数据字典） |
| **与数仓的关系** | 原始信号表、聚合/统计表（如每分钟平均车速）、事件表（如急刹车、变道）；与座舱行为做关联分析 |
| **清洗要点** | 丢包与插补、异常值（物理范围、突变）检测、时间对齐与重采样；单位与量纲统一 |

### 1.2 特征提取与标注（详细）

#### 1.2.1 标注类型与规范

| 知识点 | 说明 |
|--------|------|
| **情绪标注** | 离散标签（高兴/中性/疲劳/愤怒等）或连续维度（效价、唤醒度）；标注粒度：语句级/片段级/会话级 |
| **手势/动作标注** | 手势类型、起止时间、是否有效触发；与 IVI 控件的对应关系 |
| **驾驶行为标注** | 分心、疲劳、危险动作（如看手机）；常与 DMS 检测结果做对照与校验 |
| **标注规范文档** | 标签体系、边界 case 说明、多人标注一致性要求；版本与迭代历史 |
| **标注工具与格式** | 常见格式 JSON/COCO/自定义；工具导出与数仓表结构（样本 ID、标注结果、标注员、版本） |

#### 1.2.2 特征表设计

| 知识点 | 说明 |
|--------|------|
| **样本与特征分离** | 样本主表（样本 ID、数据路径、时间、车辆/用户维度）；特征表（样本 ID、特征名、特征值或向量） |
| **多模态对齐** | 同一会话/时间段内，视觉、语音、文本、车辆信号用统一时间或会话 ID 关联 |
| **特征版本** | 特征管线版本、模型版本、标注版本；可追溯某次训练使用的数据与特征快照 |
| **离线与实时** | 训练用离线特征表（T+1 或按批次）；线上推理用实时特征或近线特征（延迟要求） |

#### 1.2.3 与算法协作流程

| 知识点 | 说明 |
|--------|------|
| **需求澄清** | 算法提供：任务定义、所需模态、标注规范、正负样本比例；数仓/数据提供：数据范围、可获取的字段与延迟 |
| **数据供给** | 按分区/批次导出训练集、验证集；格式（TFRecord/Parquet/CSV）、路径与权限 |
| **标注闭环** | 标注平台 → 标注结果表 → 质量校验 → 反馈标注方；坏例分析与规则迭代 |

---

### 1.3 数据质量与标注质量（详细）

| 知识点 | 说明 |
|--------|------|
| **标注一致性** | Kappa/Fleiss 等一致性指标；多人标注与仲裁规则；与清洗规则中的“去重/主数据”思路一致 |
| **噪声与异常** | 错误标签、边界模糊样本的识别与剔除或二次标注；与数据清洗中的异常值处理衔接 |
| **样本平衡** | 各类别样本量、采样策略（过采样/欠采样）；在数仓层可做统计与分层抽样 |
| **与数据清洗的衔接** | 原始数据清洗（格式、空值、对齐）→ 标注 → 标注结果清洗（一致性、异常）→ 特征表；清洗日志与质量报告可统一输出 |

## 二、大模型与数据服务

### 2.1 大模型基础（详细）

#### 2.1.1 核心概念

| 知识点 | 说明 |
|--------|------|
| **预训练** | 在海量文本/多模态数据上学习通用表示；数据质量与规模直接影响基座能力 |
| **微调（Fine-tuning）** | 在领域数据（如座舱对话、车控指令）上继续训练；数据需与业务分布一致、标注准确 |
| **Prompt** | 输入给模型的指令或上下文模板；数据侧可提供“标准问-标准答”或示例，用于 few-shot 设计 |
| **RAG（检索增强生成）** | 从知识库/文档中检索相关片段，拼入 Prompt；数据侧负责知识库构建、更新与检索数据源 |

#### 2.1.2 数据如何影响模型效果

| 知识点 | 说明 |
|--------|------|
| **数据规模与覆盖** | 覆盖场景全、长尾 case 充足，模型更稳；数仓可做场景分布统计与缺口分析 |
| **数据质量** | 噪声、错误标注会拉低效果；清洗与标注质检是数据侧对模型效果的直接贡献 |
| **数据新鲜度** | 策略与话术更新后，需同步更新训练/检索数据；数仓可支持按时间分区与增量更新 |
| **与产品/算法的讨论话术** | 能说清：我们提供哪些表、更新频率、质量指标；模型效果不好时，可从数据覆盖、标注、时效三方面排查 |

---

### 2.2 MCP（Model Context Protocol）（详细）

#### 2.2.1 协议目标与角色

| 知识点 | 说明 |
|--------|------|
| **MCP 是什么** | 一种标准化协议，让大模型能够发现并调用“上下文来源”：数据、知识库、工具、服务等 |
| **核心角色** | Server：暴露资源（文件、数据库、API）；Client：大模型或 Agent；通过统一协议通信 |
| **与数据的关系** | 数据管道产出的“可被模型使用的数据”可通过 MCP Server 暴露（如知识库文档、实时状态、用户记忆摘要） |

#### 2.2.2 数据/知识库/服务如何通过 MCP 暴露

| 知识点 | 说明 |
|--------|------|
| **资源类型** | 文件与文档（说明书、FAQ）、结构化数据（表/视图）、实时能力（查询接口、推送） |
| **数据侧要做的事** | 将数仓或实时链路产出的数据，封装成 MCP 可识别的资源：如“用户偏好表”“最近行程摘要”“车型知识条目”；定义读取接口、权限与刷新策略 |
| **与现有数据管道的关系** | 离线表 → 定期同步到 MCP 可读存储；实时数据 → 通过 MCP Server 封装查询或推送；需考虑延迟、权限与脱敏 |

#### 2.2.3 数据侧适配要点（面试可答）

| 知识点 | 说明 |
|--------|------|
| **数据格式** | MCP 常用 JSON/文本；数仓表导出或 API 返回需满足协议约定的 schema |
| **更新与一致性** | 知识库类：T+1 或事件驱动更新；用户级数据：近实时或按会话更新；需与产品约定 SLA |
| **权限与脱敏** | 仅暴露模型/Agent 所需字段；敏感字段脱敏或不出域；审计与访问日志 |

---

### 2.3 Data Agent 支持（详细）

#### 2.3.1 Agent 所需的数据接口类型

| 知识点 | 说明 |
|--------|------|
| **记忆系统** | 用户习惯、历史偏好、上次对话摘要；数据形态：键值或结构化记录；需要读写、过期与聚合（如“最近 7 天偏好”） |
| **场景化数据服务** | 如“识别儿童入睡自动调音量”：需儿童检测结果、座椅/OMS 状态、当前音量与策略表；数据侧提供事件流与策略配置表 |
| **通用数据接口** | 车辆状态、行程信息、导航/音乐上下文；接口形式：REST/GraphQL 或通过 MCP 暴露 |

#### 2.3.2 数据新鲜度、权限与脱敏

| 知识点 | 说明 |
|--------|------|
| **新鲜度** | 实时类（当前车速、位置）需低延迟；记忆类可分钟级；知识库可 T+1；与产品一起定义 SLA |
| **权限** | 按角色/场景限制可见数据范围；车内 Agent 通常仅本车本用户，需与账号/车机绑定 |
| **脱敏** | 姓名、手机、地址等不直接给模型；可给脱敏或聚合结果（如“常去区域”而非精确地址） |

#### 2.3.3 参与需求讨论时可提的问题

| 知识点 | 说明 |
|--------|------|
| **数据范围** | 需要哪些实体（用户、车、行程、内容）？时间范围与粒度？ |
| **读写模式** | 仅读还是需写入（如用户确认的偏好）？一致性要求？ |
| **延迟与可用性** | 接口 P99 延迟、可用性目标；与数仓/实时链路能否满足 |

## 三、智能座舱与车联网领域

### 3.1 座舱架构（详细）

#### 3.1.1 IVI（In-Vehicle Infotainment，信息娱乐系统）

| 知识点 | 说明 |
|--------|------|
| **职责** | 导航、音乐/电台、视频、电话、设置、应用生态（如第三方 App）；部分承担语音助手入口 |
| **数据产出** | 用户操作日志（点击、搜索、播放）、导航请求与结果、媒体播放状态、设置项变更、应用使用时长与崩溃等 |
| **与数仓** | 行为事件表（事件名、时间、参数）、会话/日表统计；用于画像、推荐与体验分析 |

#### 3.1.2 DMS（Driver Monitoring System，驾驶员监控系统）

| 知识点 | 说明 |
|--------|------|
| **职责** | 监测驾驶员状态：疲劳、分心、危险动作（如打电话、未系安全带）；部分与 ADAS/自动驾驶联动 |
| **数据产出** | 图像/视频（经脱敏或仅特征）、检测结果（疲劳等级、视线方向、是否闭眼等）、告警事件 |
| **与数仓** | 告警事件表、统计（如疲劳报警次数/时长）；与车辆状态、行程关联做安全分析 |

#### 3.1.3 OMS（Occupant Monitoring System，乘员监控系统）

| 知识点 | 说明 |
|--------|------|
| **职责** | 监测舱内乘员：人数、位置、儿童/成人、是否系安全带、遗留物检测等 |
| **数据产出** | 检测结果（人数、座位占用、儿童座椅状态）、告警（遗留儿童/宠物）、可选图像（需严格脱敏） |
| **与数仓** | 事件表、与 IVI 策略联动（如儿童入睡调音量）所需的状态与事件流 |

#### 3.1.4 架构简图（可手绘面试）

```
座舱域
├── IVI     → 行为、媒体、导航、设置
├── DMS     → 驾驶员状态、告警
├── OMS     → 乘员状态、告警
└── 语音/Agent → 对话、指令、记忆
         ↓
    数据接入 / 清洗 / 数仓
         ↓
    分析、模型、MCP/Agent 服务
```

### 3.2 车联网数据（详细）

#### 3.2.1 数据形态

| 知识点 | 说明 |
|--------|------|
| **时序数据** | 按时间上报的指标：车速、转速、电量、胎压、位置等；上报频率从秒级到分钟级 |
| **事件型数据** | 点火、熄火、故障码、碰撞、OTA 状态变更等；带时间戳与属性 |
| **常见埋点** | 页面曝光、按钮点击、功能使用时长、错误日志；与 IVI/DMS/OMS 的埋点规范一致 |

#### 3.2.2 常见字段与含义（理解即可）

| 知识点 | 说明 |
|--------|------|
| **车辆维度** | 车架号（VIN）、车型、品牌、软件版本 |
| **时间与行程** | 上报时间、行程 ID、点火周期；用于关联与去重 |
| **业务字段** | 依业务而定：导航的起终点、音乐的歌曲 ID、DMS 的告警类型等 |
| **数据字典** | 每类数据源应有字段说明、枚举值、单位；数仓建表与清洗时需对齐 |

#### 3.2.3 数据接入与清洗规则讨论

| 知识点 | 说明 |
|--------|------|
| **接入方式** | 车端 →  TSP/云平台 → Kafka 或文件 → 数仓；需明确延迟、重试与去重策略 |
| **清洗规则** | 无效数据过滤（如车辆未点火时的某些信号）、重复上报去重、时间戳修正与分区 |
| **与座舱的关联** | 通过车辆 ID、时间或行程 ID 与座舱行为、DMS/OMS 事件做联合分析 |

---

### 3.3 汽车网络协议（详细）

#### 3.3.1 CAN（Controller Area Network）

| 知识点 | 说明 |
|--------|------|
| **基本概念** | 车内 ECU 之间通信的主流总线；帧结构：ID、数据长度、数据域（通常 8 字节）；广播式，多节点收发 |
| **与座舱数据的关系** | 部分座舱相关信号走 CAN（如车速、转向、车门状态）；若数仓需接入“车辆信号”，会涉及 CAN 解析后的信号表 |
| **数据字典** | DBC 文件定义信号名、起始位、长度、缩放与偏移、单位；解析后得到物理值入表 |

#### 3.3.2 LIN（Local Interconnect Network）

| 知识点 | 说明 |
|--------|------|
| **基本概念** | 低成本、低速、主从结构；常用于座椅、车窗、雨刮等从节点 |
| **与座舱数据的关系** | 座椅状态、车窗等可能经 LIN 汇总后经 CAN 或网关上传；数仓更多接触的是已解析后的状态表 |

#### 3.3.3 数仓/数据工程师需要掌握的程度

| 知识点 | 说明 |
|--------|------|
| **能理解协议文档** | 知道 DBC/数据字典在说什么；能对应“信号名 → 数仓字段” |
| **能参与数据字典维护** | 新增信号、单位与枚举值；与嵌入式/网关团队对齐命名与含义 |
| **深入优先项** | 若岗位涉及信号解析或实时 CAN 接入，再学帧结构、多路复用与解析流水线 |
